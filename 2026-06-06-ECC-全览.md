## 第一部分：Rules — 20 套语言的编码宪法

Rules 是 ECC 注入到每个会话的系统级上下文。它们不是建议，是约束。ECC 的规则体系按三层组织：

1. `common/` — 所有语言通用的规则
2. 19 个语言/框架目录 — 每个 5-7 个规则文件
3. `zh/` — 中文翻译版 common rules

### common/（10 个文件）— 跨语言基础设施

| 文件 | 约束内容 |
|------|----------|
| `agents.md` | Agent 编排规范——何时用 planner、何时用 architect、何时用 code-reviewer。定义了"写完代码立即审查、复杂功能先规划、安全敏感代码必须过 security-reviewer"的自动化触发规则 |
| `code-review.md` | 代码审查标准——审查前检查清单（功能正确性、安全性、性能、测试覆盖、代码质量），强制性审查触发条件（≥200 行变更、新增 API 端点、认证/授权代码、数据库迁移），以及审查后不良行为追踪 |
| `coding-style.md` | 通用编码风格——不可变性铁律（永远创建新对象，永不修改已有对象）、KISS/DRY/YAGNI 原则、函数 <50 行、文件 <800 行、嵌套深度 ≤4 层、错误处理每层都要有 |
| `development-workflow.md` | 全功能开发流程——Plan → TDD → Review → Capture Knowledge → Commit 的五阶段流程，每个阶段的进出条件和 Agent 调用规则 |
| `git-workflow.md` | Git 操作规范——Conventional Commits 格式、分支命名规则、PR 准备工作清单、commit message 自动生成原则 |
| `hooks.md` | Hook 使用规范——PreToolUse（工具调用前验证）、PostToolUse（工具调用后自动检查）、Stop（会话结束时的会话摘要和记忆持久化） |
| `patterns.md` | 通用架构模式——"实现前先找已有模式"原则、Skeleton 项目选择策略（用并行 Agent 评估模板的安全性/可扩展性/相关性，选最好的克隆）、Repository 模式、一致的 API 响应格式 |
| `performance.md` | 性能与模型选择——Haiku 4.5（Sonnet 90% 能力，3x 成本节省）用于轻量 Agent、Sonnet 4.6 用于主力开发、Opus 4.5 用于架构决策。上下文窗口管理：大型重构避让最后 20% 空间 |
| `security.md` | 安全基线——提交前 8 项强制检查（无硬编码密钥、输入验证、SQL 注入防护、XSS 防护、CSRF、认证验证、限流、错误消息不泄露敏感信息）。安全事件响应协议：STOP → security-reviewer → 修复 CRITICAL → 轮换密钥 → 审查全库 |
| `testing.md` | 测试铁律——最低 80% 覆盖率。三种测试类型均强制要求：单元测试、集成测试、E2E 测试。TDD 六步强制流程（RED → FAIL → GREEN → PASS → REFACTOR → 80%+）。测试失败时的排障链路 |

### 19 个语言/框架规则包

每个语言目录包含相同的 5-7 个文件结构，但内容完全针对该语言/框架定制：

**通用结构（所有语言共享）：**
- `coding-style.md` — 语言特定的编码风格（如 Go 的 error handling 模式、Rust 的 ownership 规则、Python 的 type hints）
- `hooks.md` — Hook 在该语言项目中的最佳触发条件
- `patterns.md` — 该语言生态的惯用设计模式（如 Kotlin 的协程模式、React 的 hooks 规则）
- `security.md` — 该语言的特定安全风险（如 Python 的 `pickle` 反序列化、JavaScript 的 prototype pollution）
- `testing.md` — 该语言的测试框架选择和最佳实践

**有额外文件的语言：**
- `python/` 额外有 `fastapi.md` — FastAPI 的异步模式、依赖注入、Pydantic 模型、OpenAPI 文档生成
- `web/` 额外有 `design-quality.md` 和 `performance.md` — 前端设计质量标准（色彩、排版、动画）和 Web 性能（Core Web Vitals、bundle size、lazy loading）

每个规则文件都有 `paths` frontmatter 字段，定义了该规则在什么文件路径下生效（如 `paths: ["**/*.ts", "**/*.tsx"]`）。这意味着一个 Java 项目的 Agent 会话不会加载 React 规则——只有匹配文件路径的规则才会注入。

### 2.2 工程方法技能（~30 个）

这些 skill 定义了"怎么写代码"而不是"写什么代码"。

- `tdd-workflow` — 强制性 TDD 流程：写测试先于实现，80%+ 覆盖率要求，测试失败的排障链路
- `agentic-engineering` — Agent 驱动的工程方法：eval-first 执行模型、15 分钟任务单元分解、按复杂度路由模型（Haiku=简单/Sonnet=实现/Opus=架构）
- `ai-regression-testing` — AI 开发的回归测试策略：沙盒模式 API 测试、自动快照对比
- `verification-loop` — 综合验证系统：检查点式 vs 连续式评估、评分器类型、pass@k 指标
- `search-first` — "搜索优先"工作流：在写代码之前先搜索已有工具、库和模式，用 Context7 查文档、用 Exa 搜方案
- `context-budget` — 上下文预算审计：分析 Agent/Skill/MCP/Rules 的上下文消耗，识别膨胀、冗余和优化机会
- `token-budget-advisor` — Token 预算顾问：按任务分配合适的上下文配额
- `cost-tracking` — 成本追踪：从本地成本数据库追踪 Token 使用量、花费和预算
- `cost-aware-llm-pipeline` — 成本感知的 LLM 流水线：按任务复杂度自动选择最经济的模型
- `benchmark` — 性能基准：测量基线、检测 PR 前后的性能回归、对比技术栈方案
- `benchmark-optimization-loop` — 基准优化循环：跑 benchmark → 分析瓶颈 → 优化 → 再跑 benchmark 的自动化闭环
- `parallel-execution-optimizer` — 并行执行优化器：并行 Agent、批量工具调用、隔离工作树、结果合并
- `data-throughput-accelerator` — 数据吞吐加速：大数据摄入/回填/导出/ETL/数仓加载的并行策略
- `latency-critical-systems` — 延迟敏感系统：尾延迟优化、预热策略、资源预留

## 第四部分：Commands — 79 个斜杠命令

ECC 的路线图明确标注 `commands/` 是 legacy surface，长期方向是 skill-first。但 79 个命令仍在活跃使用，涵盖了从规划到部署到运维的完整工作流。

### 开发流程（9 个）

`/plan` — 重述需求，评估风险，创建逐步实现计划。**等待用户确认后才碰代码**。不猜测用户意图，不确定的地方会停下来问。

`/plan-prd` — 交互式 PRD 生成器。以问题优先、假设驱动的方式生成产品需求文档，通过反复问答澄清需求。

`/prp-plan` — 创建全面的功能实现计划，含代码库分析和模式提取。是 PRP（Plan → Implement → Commit → PR）流程的第一步。

`/prp-implement` — 执行实现计划，带严格的验证循环。实现过程中持续检查代码是否符合 plan 中定义的规范。

`/prp-commit` — 用自然语言快速提交。描述要提交什么内容，Agent 自动选择文件、生成 commit message 并执行提交。

`/prp-pr` — 从当前分支创建 GitHub PR，发现模板、分析变更、推送。等价于 `/pr` 但在 PRP 流程中的命名统一版本。

`/pr` — 从当前分支创建 GitHub PR。自动发现项目模板，分析提交历史生成综合摘要，包含测试计划。

`/feature-dev` — 引导式功能开发。七阶段流水线（Discovery → Explore → Clarify → Architect → Implement → Review → Summary），使用专门的 code-explorer、code-architect、code-reviewer Agent。

`/aside` — 在不中断当前任务上下文的情况下回答一个快速问题。回答问题后自动恢复原任务。适合"顺便问一下"的场景。

### 代码质量与安全（8 个）

`/code-review` — 代码审查，支持两种模式：审查本地未提交的变更，或传入 PR 编号/URL 审查 GitHub PR。调用 language-specific reviewer Agent。

`/build-fix` — 检测项目构建系统，增量修复构建/类型错误。只做最小安全改动，每次修复后验证。

`/refactor-clean` — 安全地识别和移除死代码（未使用的变量、函数、导入、文件），每次清理后验证功能不受影响。

`/test-coverage` — 分析覆盖率，识别缺口，生成缺失的测试，向目标阈值（80%+）推进。

`/security-scan` — 运行 AgentShield 安全扫描。覆盖 agent、hook、MCP、权限、密钥五个攻击面，共 1282 个测试用例和 102 条检测规则。

`/quality-gate` — 运行 ECC 质量管道（lint、测试、安全扫描、类型检查），报告文件或项目范围内的不合规项和修复步骤。

`/harness-audit` — 对仓库运行确定性 Agent harness 审计，返回按优先级排序的评分卡。检查 agent 配置质量、skill 覆盖率、rules 是否需要更新。

`/model-route` — 根据当前任务的复杂度、风险和预算，推荐最佳模型层级（Haiku/Sonnet/Opus）。

`/hookify-help` — 获取 hookify 系统的帮助文档。覆盖规则语法、匹配模式、优先级顺序。

`/hookify-list` — 列出所有已配置的 hookify 规则。显示每条规则的匹配模式、触发时机和当前状态（启用/禁用）。

### 会话管理（5 个）

`/save-session` — 将当前会话状态保存到 `~/.claude/session-data/` 的带日期文件中，包含完整上下文，后续可通过 `/resume-session` 恢复。

`/resume-session` — 从 `~/.claude/session-data/` 加载最近的会话文件，带完整上下文恢复工作。适合跨天的长任务。

`/sessions` — 管理 Claude Code 会话历史、别名和会话元数据。可以给重要会话打标签、添加备注。

`/checkpoint` — 在运行验证检查后创建、验证或列出工作流检查点。每个检查点包含当前的测试通过状态和代码快照引用。

`/prune` — 删除超过 30 天未提升（promote）的 pending instincts。保持 instinct 数据库干净，避免过期模式污染新会话。

### 项目管理与操作（5 个）

`/projects` — 列出已知项目及其 instinct 统计信息。显示每个项目有多少 learned patterns、哪些是最常触发的规则。

`/project-init` — 检测项目技术栈，生成 ECC 入职计划（dry-run）。使用仓库的安装 manifest 和技术栈映射，推荐应安装的 rules/skills/agents。

`/promote` — 将项目级别的 instincts 提升到全局范围。当一个 pattern 在多个项目中都有用，提升后所有项目共享。

`/pm2` — 分析项目并生成 PM2 服务管理命令。自动检测前端、后端、数据库服务，生成启动/停止/重启/日志查看命令。

`/setup-pm` — 配置首选包管理器（npm/pnpm/yarn/bun）。ECC 会记住这个偏好并在所有后续操作中使用。

### 维护与文档（6 个）

`/update-docs` — 从真实来源文件（scripts、schemas、routes、exports）同步文档。不凭空生成——从代码中提取实际接口和结构。

`/update-codemaps` — 扫描项目结构，生成 token-lean 架构代码地图。为大型代码库提供紧凑的结构概览，不消耗大量上下文 token。

`/auto-update` — 拉取最新 ECC 仓库变更并重新安装当前管理的 targets。保持 ECC 本身是最新版本。

`/skill-create` — 分析本地 git 历史，提取编码模式，生成 SKILL.md 文件。是 Skill Creator GitHub App 的本地版本。

`/skill-health` — 显示 skill 组合健康度仪表盘。含图表和分析——哪些 skill 经常被触发、哪些从未使用、哪些需要更新。

`/ecc-guide` — 导航 ECC 当前的 agents、skills、commands、hooks、安装 profiles 和文档。从活跃的仓库表面实时读取，不是静态帮助文档。

### 循环与自主执行（3 个）

`/loop-start` — 启动一个托管的自主循环模式。带安全默认值（最大迭代次数、超时、成本上限）和显式停止条件。由 `loop-operator` Agent 监控。

`/loop-status` — 检查活跃循环的状态、进度、失败信号和建议的干预措施。循环出问题时不需要手动 kill——先看状态再决定。

`/santa-loop` — 对抗性双审查收敛循环。两个独立的模型审查者必须都批准后代码才能发布。基于"Santa Method"（列清单，检查两遍）。

### 集成与专项（5 个）

`/jira` — 获取 Jira ticket，分析需求，更新状态或添加评论。使用 `jira-integration` skill，通过 MCP 或 REST API 连接。

`/marketing-campaign` — 规划和执行完整营销活动。接收产品简介，返回定位策略、着陆页文案、邮件序列、社交媒体帖子和广告变体。

`/cost-report` — 从本地成本追踪 SQLite 数据库生成 Claude Code 成本报告。按项目、模型、时间段分析 Token 使用和花费。

`/security-scan` —（已在代码质量一节列出，此处不重复）

`/ecc-guide` —（已在维护一节列出，此处不重复）

## 第五部分：ECC 2.0 — Rust 控制平面（alpha）

`ecc2/` 是 ECC 最激进的部分——一个用 Rust 写的 Agent 运行时：

```bash
ecc dashboard     # 启动 TUI 控制面板
ecc start         # 启动 Agent 会话
ecc sessions      # 列出所有会话
ecc status        # 状态快照（可导出 markdown）
ecc stop          # 停止会话
ecc resume        # 恢复暂停的会话
ecc daemon        # 后台守护进程
```

`ecc status --markdown --write status.md` 将本地状态转化为可移植的交班文档：

- 就绪状态（哪些 rules/skills/agents 已安装且健康）
- 活跃会话（当前运行的 Agent、进度、资源消耗）
- Skill-run 健康度（哪些 skill 有失败的运行记录）
- 安装健康度（哪些组件版本不匹配）
- 待处理治理事件（需要人工决策的事项）
- 关联工作项（Linear tickets、GitHub issues/PRs、handoff 记录）

这不是一个"配置集合"的扩展——这是在构建一个 **Agent 操作系统的运行时**。桌面 GUI（`ecc_dashboard.py`）提供了可视化浏览 Agents、Skills、Commands、Rules 的界面，支持搜索过滤和暗色/亮色主题切换。

## 第六部分：跨平台适配

ECC 同一套定义可以运行在 7 个平台上：

| 平台 | 适配方式 |
|------|----------|
| **Claude Code** | 原生——完整支持 plugin + manual 两条安装路径 |
| **Codex CLI** | `AGENTS.md` 驱动——独立 Codex 安装器和配置文件 |
| **Cursor** | `.cursor/rules` 目录注入 |
| **OpenCode** | 全插件支持（12 agents、24 commands、16 skills、20+ 事件 hooks） |
| **Gemini CLI** | `.gemini/` 配置目录 |
| **Zed** | `.zed/` 配置目录 |
| **GitHub Copilot** | `.github/copilot-instructions.md` |

适配策略不是"抽象出一套中间 DSL 再翻译"——那样会丧失每个 harness 的原生能力。而是**每个 harness 有自己适配的配置层**（`.claude/`、`.codex/`、`.cursor/`、`.opencode/` 等），核心 Agent prompt 和 skill 逻辑在整个仓库中保持唯一真实来源（single source of truth）。除 Rules 之外的组件（manifests、plugin.json、npm 包）通过 `manifests/` 目录管理跨 harness 分发。

## 五条贯穿始终的设计决策

### 1. Skill-first，Command 退役

`skills/` 是 canonical workflow surface，`commands/` 是 legacy shim。Skill 可以自动加载（Agent 按 `description` 判断何时需要），Command 必须手动输入 `/` 调用。当系统有 249 个 workflow 时，**隐式按需加载 > 显式手动调用**。

### 2. 语言第一性

ECC 的 Agent 和 Rule 体系是按语言第一性组织的。好的代码审查、好的构建修复、好的安全审计是语言特定的。一个 Kotlin reviewer 和一个 Go reviewer 审查同一段逻辑，结论可能完全相反——因为"好代码"的定义在这两个生态里不同。Rules 的 `paths` frontmatter 确保 Java 项目不会加载 React 规则，避免上下文污染。

### 3. 选择性安装

249 个 skill 不可能全装。ECC 提供渐进式安装路径：
- `--profile minimal` — 核心规则，无 hooks
- `--profile core --without baseline:hooks` — 不要自动拦截
- `npx ecc consult "mlops training"` — 先搜索匹配组件再装
- 规则按语言选择性复制（`cp -R rules/typescript ~/.claude/rules/ecc/`）

### 4. 状态持久化

ECC 的 hooks 包含 `memory-persistence`——跨会话的 structured recording with session adapters。ECC 2.0 的 `ecc status` 把运行时状态变成可读、可审计、可交班的文档。

### 5. 开源但可持续

MIT 许可，OSS 永远免费。ECC Pro 是托管 GitHub App（$19/seat/mo），面向私有仓库的 PR 审计和高级安全功能。两个 npm 包（`ecc-universal` 和 `ecc-agentshield`）提供了独立的发布渠道。
