# 从 ECC 偷设计：高级开发者如何改进自己的 AI 编码工作流

ECC（Everything Claude Code）是 182K+ stars、249 个 skill、63 个 agent、79 个 command 的巨型项目。大多数人装完 ECC 就用它提供的命令和 skill，这当然可以。

但这个项目更大的价值在于：**它是一份 10+ 个月密集实战中沉淀出来的 AI 编码工作流设计教科书**。你不需要全盘接受它——你可以偷它的设计模式，嫁接到你自己的工作流上。

仓库：[affaan-m/ECC](https://github.com/affaan-m/ECC) ⭐182K+ 🔀28K+ 📄MIT 📅2026-06-07

这篇文章不是 ECC 的使用教程（那在上一篇）。这篇是站在一个已经有自己工作流的高级开发者角度，逐个拆解 ECC 中值得偷的设计模式——哪些可以直接用、哪些需要改造、哪些只适合它的规模不适合个人。

---

## 对高级开发者有什么用

你已经有了自己的 CLAUDE.md、自己写的几个 skill、几个 hook。你不需要 ECC 的 249 个 skill——你大概只需要其中 20 个。你需要的是：**理解 ECC 为什么这样设计**，然后把设计原则提取出来，应用到自己的系统里。

具体来说，ECC 解决了六个你已经遇到但可能还没系统化解决的问题：

1. **上下文窗口管理**：skill 和 MCP 太多，上下文不够用，输出质量下降
2. **agent 质量控制**：agent 写完代码你不知道好不好，只能肉眼检查
3. **自主循环的安全边界**：想让 agent 自动跑一晚上，但怕它搞坏代码库
4. **决策质量**：agent 总是同意你说的，缺少对抗性审查
5. **知识积累**：每次会话结束，调试经验和架构决策就丢了
6. **工作流的可评估性**：改了一个 prompt，不知道是变好了还是变差了

## 模式 1：GateGuard——用事实强制取代自我评估

这是 ECC 中最有实验数据支撑的设计。

### 问题

LLM 的自我评估不工作。你问 Claude "你刚才的代码有没有违反项目规范？"，它永远回答"没有"。这是经过实验验证的——ECC 的 gateguard skill 原文说：

> "LLM self-evaluation doesn't work. Ask 'did you violate any policies?' and the answer is always 'no.' This is verified experimentally."

### ECC 的方案

gateguard 是一个 PreToolUse hook，在 agent 调用 `Edit`/`Write`/`Bash` 之前拦截。它不是问"你确定吗？"——它**强制 agent 去调查具体事实**：

```
阶段 1: DENY  — 阻止第一次 Edit/Write/Bash 调用
阶段 2: FORCE — 告诉模型需要收集哪些事实
         （"列出所有 import 这个模块的文件"、"读取 schema 定义"）
阶段 3: ALLOW — 事实收集完毕后才允许重试
```

核心洞察：**调查行为本身产生了上下文，而这个上下文改变了输出**。自我评估是空转——agent 不需要做任何事就能说"我确认过了"。但强制调查需要 agent 真的去跑 Grep、Read，拿到数据后才能继续。

两组 A/B 测试数据：同一 agent、同一任务，有 gate 比无 gate 的 output quality 高 +2.25 分。

### 偷到自己工作流中

你不需要 ECC 的完整 gateguard 实现。核心思想可以用一个 20 行的 hook 实现：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "tool_name == 'Edit' || tool_name == 'Write'",
        "hooks": [{
          "type": "command",
          "command": "echo 'Before editing, list all files that import the module you are about to change. Run Grep first.'"
        }]
      }
    ]
  }
}
```

这个 hook 的效果：每次 agent 要改代码，先被提醒去查依赖关系。不需要 block（那太重了），一个 stderr 提醒就够了。

**适用场景**：任何你发现 agent "自信地犯错误"的领域。它以为自己知道，所以不去查——gate 强制它去查。

**不适用场景**：简单的 bug fix、单文件改动。gate 的代价是多一轮工具调用（token + 时间），对于简单任务不值得。

## 模式 2：Context Budget——上下文窗口的预算管理

### 问题

你的 Claude Code 会话有 200K token 上下文。但你装了 10 个 MCP server（每个 500 token 工具定义）、20 个 skill（平均 2000 token）、5 个 rule 文件（平均 500 token）——还没开始干活，50K token 就没了。

ECC 的 The Shortform Guide 原话：

> "Your 200k context window before compacting might only be 70k with too many tools enabled. Performance degrades significantly."
>
> "Rule of thumb: Have 20-30 MCPs in config, but keep under 10 enabled / under 80 tools active."

### ECC 的方案

ECC 有一个专门的 `context-budget` skill，做四件事：

1. **Inventory**：扫描所有加载组件，估算 token 消耗
   - 每个 agent 文件 >200 行的标记为"heavy"
   - 每个 skill 文件 >400 行的标记
   - 每个 MCP server 的工具数 × 500 token
   - CLAUDE.md 合计 >300 行的标记

2. **Classify**：每个组件归入三类
   - Always needed（核心，保留）
   - Sometimes needed（按需加载，不要默认启用）
   - Never needed（删除或归档）

3. **Recommend**：按 token 节省量排序的优化建议

4. **Monitor**：跟踪优化前后的上下文使用量变化

### 偷到自己工作流中

你不需要 ECC 的完整 context-budget skill。核心原则是：**把上下文窗口当成内存预算来管理**。

具体操作：

**第一步：审计。** 跑一次 `cat ~/.claude/settings.json` 看当前加载了什么。每个 MCP server 的工具定义大约 500 token/tool。如果你有 5 个 MCP server 各 20 个工具，那就是 50K token——你上下文的 25% 在还没开始干活时就没了。

**第二步：分级。** 把你的 skill 和 MCP 分三级：
- 每个会话都用的（比如 TypeScript LSP、安全 hook）→ 保留
- 特定任务才用的（比如 Django skill、Playwright MCP）→ 按需启用
- 三个月没用过的 → 删掉

**第三步：定期复查。** 每次装了新的 MCP 或 skill 后，问自己："我上一次用这个是什么时候？"如果用不上，禁用。

**关键数字**：80 个活跃工具是 ECC 推荐的上限。超过这个数字，agent 的输出质量明显下降——不是因为它"记不住"，而是因为工具选择变成了组合爆炸问题。

## 模式 3：Council——对抗性决策框架

### 问题

你让 Claude 帮你做架构决策（"应该用 monorepo 还是 polyrepo？"），它倾向于同意你——你说倾向 monorepo，它就给你列 monorepo 的优势。这是"对话锚定"（conversational anchoring）：agent 被当前对话上下文锁定了，无法产生真正的对抗性思考。

### ECC 的方案

ECC 的 `council` skill 召集四个"顾问"做结构化对抗：

| 角色 | 视角 |
|------|------|
| Architect | 正确性、可维护性、长期影响 |
| Skeptic | 挑战前提、简化、打破假设 |
| Pragmatist | 交付速度、用户影响、运维现实 |
| Critic | 边界情况、下行风险、失败模式 |

关键设计：**三个外部声音必须作为全新 subagent 启动，只给它们"问题和相关上下文"，不给完整的对话历史**。这就是反锚定机制（anti-anchoring mechanism）。

为什么不给完整对话？因为完整对话中包含了你的倾向、你的前期论证、你的情绪——所有这些东西都会让 subagent 倾向于同意你。只给问题本身，subagent 才能独立思考。

### 偷到自己工作流中

你不需要 ECC 的完整 council skill。核心思想用三个 subagent 就能实现：

当你面对一个有多种可行方案的决策时，启动三个 subagent，每个只给决策问题本身（不给前面的对话）：

```
Subagent 1（Skeptic）："我正在决定 X vs Y。约束条件是 A, B, C。
请挑战我的前提假设——为什么我可能根本不该在这两个选项中选？"

Subagent 2（Pragmatist）："同上问题。
请从交付速度和运维成本的角度分析。"

Subagent 3（Critic）："同上问题。
请列出每个方案最可能失败的三种方式。"
```

**适用场景**：架构决策、技术选型、ship vs hold 的判断。任何"有多个可信方案且没有明显赢家"的情况。

**不适用场景**：代码审查（用 code-reviewer）、功能规划（用 planner）、事实性问题（直接回答）。ECC 的 council skill 明确列出了不该用的场景——把 council 用在错误的场景是浪费 token。

## 模式 4：自主循环的安全边界

### 问题

你想让 agent 自动跑一个循环——比如每 30 分钟跑一次测试，失败了自动修复，直到全部通过。但你怕：它可能陷入死循环，可能反复犯同一个错误，可能在你不知道的情况下烧掉大量 token。

### ECC 的方案

ECC 有两种循环模式，安全设计截然不同：

**1. Sequential Pipeline（`claude -p`）**——最简单的循环：

```bash
while true; do
  claude -p "运行测试，修复失败的用例，提交" 
  sleep 1800
done
```

安全边界靠外层 shell 控制——agent 本身不知道自己在循环中。

**2. loop-operator agent**——有内建安全机制的循环：

ECC 的 loop-operator agent 有明确的**停止条件和升级条件**：

```
停止条件：
- 任务完成（所有测试通过）
- 达到最大迭代次数
- 用户中断

升级条件（需要人类介入）：
- 连续两个 checkpoint 没有进展
- 相同错误堆栈的重复失败
- 成本超出预算窗口
- merge 冲突阻塞队列
```

loop-operator 的工作流：
1. 从显式的循环模式和模式启动
2. 跟踪进度 checkpoint
3. 检测停滞和重试风暴
4. 重复失败时暂停并缩小范围
5. 只在验证通过后才恢复

**3. Ralphinho / RFC-Driven DAG**——最复杂的模式：

大型功能拆成多个并行 unit，每个 unit 有独立的 agent 在循环中迭代，通过 merge queue 协调。这是 ECC 中最接近"自主开发团队"的模式。

### 偷到自己工作流中

核心原则：**自主循环必须有外部的进度检测和强制中断机制**。agent 自己不能判断"我是不是在原地打转"——这必须从外部检测。

最简单的实现：

```bash
MAX_ITERATIONS=10
PREV_HASH=""
for i in $(seq 1 $MAX_ITERATIONS); do
  claude -p "运行测试，修复失败用例，git commit"
  
  # 进度检测：检查 git log 是否有新 commit
  CURR_HASH=$(git log --oneline -1 | awk '{print $1}')
  if [ "$CURR_HASH" == "$PREV_HASH" ]; then
    echo "No progress in iteration $i. Stopping."
    break
  fi
  PREV_HASH=$CURR_HASH
  
  # 成功检测：测试全过就退出
  if npm test 2>/dev/null; then
    echo "All tests pass. Done."
    break
  fi
done
```

**关键设计**：
- **进度检测**不是看 agent 说了什么，而是看**有没有产生新的 commit**
- **强制上限**防止无限循环
- **成功退出**条件明确

**不适用场景**：涉及生产环境的操作。自主循环只应该在 feature branch 上跑，有 git worktree 隔离。

## 模式 5：Eval-Driven Development——工作流的可评估性

### 问题

你改了一个 skill 的 prompt，或者换了一个模型版本。你怎么知道变好了还是变差了？大多数人的答案是"感觉差不多"——这不是工程。

### ECC 的方案

ECC 的 `eval-harness` skill 实现了一个完整的评估框架，核心理念是 Eval-Driven Development（EDD）：**先定义成功标准，再实现**。

三种 eval 类型：

**Capability Eval**——测试 agent 能不能做一件它之前做不到的事：
```markdown
[CAPABILITY EVAL: django-migration]
Task: 为 User 模型添加 email_verified 字段并生成迁移文件
Success Criteria:
  - 迁移文件存在且可执行
  - 字段类型为 BooleanField(default=False)
  - 没有硬编码默认值
Expected Output: 一个可运行的 Django migration 文件
```

**Regression Eval**——确保改动不破坏已有功能：
```markdown
[REGRESSION EVAL: react-component]
Baseline: commit abc123
Tests:
  - auth-component renders: PASS/FAIL
  - form validation works: PASS/FAIL
  - API integration test: PASS/FAIL
Result: X/Y passed (previously Y/Y)
```

**两种 grader**：

- **Code-Based Grader**：确定性检查（`grep`、`npm test`、`npm run build`），结果是确定的 PASS/FAIL
- **Model-Based Grader**：用另一个 Claude 实例评估开放式输出（代码质量、设计合理性），适合无法用确定性测试覆盖的场景

### 偷到自己工作流中

你不需要 ECC 的完整 eval-harness。核心原则：**每次改 prompt 或换模型，跑一组固定的测试用例**。

最简单的实现：

1. 创建 `evals/` 目录，每个 eval 是一个 markdown 文件：
   ```
   evals/
     django-api-endpoint.md    # 任务描述 + 成功标准
     react-form-validation.md
     go-error-handling.md
   ```

2. 每个 eval 包含：任务描述、成功标准（可检查的 checklist）、grader 类型（code-based 还是 model-based）

3. 改完 prompt 后，对每个 eval 跑一次，记录 pass/fail

4. 用一张表跟踪：
   ```
   | Eval                  | v1 prompt | v2 prompt | Opus 4.6 | Sonnet 4.6 |
   |-----------------------|-----------|-----------|----------|------------|
   | django-api-endpoint   | PASS      | PASS      | PASS     | FAIL       |
   | react-form-validation | FAIL      | PASS      | PASS     | PASS       |
   | go-error-handling     | PASS      | PASS      | PASS     | FAIL       |
   ```

这就是 ECC 的 eval-harness 的简化版。核心是**可量化、可复现、可对比**。

**关键数字**：ECC 的 AGENTS.md 要求"Minimum coverage: 80%"。你可以给自己定一个类似的门槛——比如 10 个 eval 中至少 8 个 PASS，否则不升级 prompt 或模型。

## 模式 6：Continuous Learning——知识的跨会话积累

### 问题

你在一个 4 小时的会话中解决了一个复杂的 bug——发现了 Django ORM 的 `select_related` 在特定 join 条件下会产生笛卡尔积。这个知识在当前会话中有用，但会话结束后就丢了。下次遇到同样的问题，agent 还是会犯同样的错误。

### ECC 的方案

ECC 的 `continuous-learning-v2` skill 是一个跨会话的知识积累系统（v1 已废弃）：

**v1（已废弃）**：用 Stop hook 在会话结束时提取模式，保存为 learned skill。问题太粗糙——提取的模式粒度太粗，质量不稳定。

**v2（当前）**：三个改进：

1. **从 Stop hook 扩展到 PreToolUse/PostToolUse 观察**——不只是会话结束时提取，而是在整个会话过程中持续观察 agent 的行为模式

2. **从完整 skill 变成原子 instinct**——不是提取一整个 skill，而是提取一个具体的"直觉"（instinct），带有置信度评分。例如：
   ```yaml
   instinct: "Django ORM 的 select_related 在多对多 join 时产生笛卡尔积"
   confidence: 0.8
   source: "session-2026-05-15-debug"
   project_scoped: true
   ```

3. **从全局存储变成项目作用域 + 全局提升**——先存在项目级别，积累足够证据后才提升到全局

### 偷到自己工作流中

核心原则：**agent 的错误是学习材料，不是噪音**。每次 agent 犯了一个你在审查时发现的错误，问自己："这个错误会再发生吗？如果会，我能不能写一条 rule 防止它？"

最简单的实现：

1. 在你的 `rules/` 目录中维护一个 `lessons-learned.md`：
   ```markdown
   # Lessons Learned
   
   ## 2026-06-01: Django ORM N+1 查询
   在使用 select_related 时，如果涉及多对多字段，
   必须使用 prefetch_related 而不是 select_related。
   select_related 在多对多场景下会产生笛卡尔积。
   
   ## 2026-06-05: React useEffect 闭包陷阱
   在 useEffect 中使用 state 变量时，必须将变量加入
   依赖数组，否则会使用过期的闭包值。
   ```

2. 每次会话结束时（或者你发现 agent 犯了一个之前犯过的错误时），追加一条。

3. 定期复查：哪些 lesson 已经被 agent 正确遵守了（可以删掉），哪些还在反复犯错（需要加强措辞或变成 hook）。

这比 ECC 的 continuous-learning-v2 简单得多，但覆盖了 80% 的价值。

## 模式 7：Hook 分层防御——即时拦截 + 深度分析 + 最后闸门

### 问题

你有一个安全检查想在 agent 写代码时执行。但安全检查有两种：快速的（正则匹配危险模式）和慢速的（LLM 审查 diff）。放在哪里？

### ECC 的方案

ECC 的安全相关设计使用了三层防御，每层用不同的扩展机制：

| 层 | 机制 | 速度 | 深度 | 时机 |
|----|------|------|------|------|
| 第一层 | PreToolUse hook（正则） | 毫秒级 | 浅（模式匹配） | 每次 Edit/Write 时 |
| 第二层 | Stop hook（LLM diff review） | 秒级 | 深（语义分析） | agent 完成一轮修改后 |
| 第三层 | PostToolUse on git commit（agent 审查） | 分钟级 | 最深（全面安全审查） | 提交前 |

这个分层来自 Claude Code 官方的 security-guidance 插件（David Dworken 设计），ECC 在自己的 hook 系统中复现了同样的模式。

### 偷到自己工作流中

这个模式可以泛化到任何"需要多层质量控制"的场景，不只是安全：

**第一层（Hook，毫秒级）**：正则匹配你已知的反模式。

```json
{
  "PreToolUse": [{
    "matcher": "tool_name == 'Write'",
    "hooks": [{
      "type": "command",
      "command": "grep -q 'console.log' \"$CLAUDE_FILE_PATH\" 2>/dev/null && echo '[WARN] console.log found. Remove before commit.' >&2 || true"
    }]
  }]
}
```

**第二层（Skill，按需调用）**：写完一个功能后，手动 `/code-review` 或 `/security-scan`。

**第三层（Hook，commit 时）**：`git commit` 前自动跑一轮完整检查。

关键取舍：第一层必须快（正则，不消耗 token），因为每次编辑都会触发。第二层和第三层可以慢（消耗 token），因为它们只在特定时机触发。

## 模式 8：Harness Optimizer——不改代码改环境

### 问题

你的 agent 输出质量不够好。直觉反应是改 prompt——但 ECC 的 harness-optimizer agent 提出了一个不同的视角：

> "Raise agent completion quality by improving harness configuration, not by rewriting product code."

翻译：**改环境比改 prompt 更有效**。

### ECC 的方案

harness-optimizer agent 的工作流：

1. 跑 `/harness-audit`，获取基线评分
2. 识别前 3 个杠杆领域（hooks、evals、routing、context、safety）
3. 提出最小的、可逆的配置变更
4. 应用变更并跑验证
5. 报告 before/after 差异

约束：
- 优先选择有可测量效果的小改动
- 保持跨平台行为一致
- 避免引入脆弱的 shell quoting
- 兼容 Claude Code、Cursor、OpenCode、Codex

### 偷到自己工作流中

在改 prompt 之前，先检查这些"环境因素"：

1. **上下文是否被浪费？** 禁用了不用的 MCP server 吗？skill 中有三个月没碰过的吗？（参考模式 2）

2. **规则是否冲突？** 你的 CLAUDE.md 和 rules 中有没有互相矛盾的指令？比如一个说"用 async/await"，另一个说"用 callback"。

3. **Hook 是否过度限制？** 你的 PreToolUse hook 是否在不必要的场景阻止了 agent？过度限制和没有限制一样影响输出质量。

4. **模型选择是否匹配任务？** 不是所有任务都需要 Opus。简单的代码审查用 Sonnet 就够了，把 Opus 留给需要深度理解的场景（架构设计、复杂 debug）。ECC 的 63 个 agent 中，大多数用 Sonnet——只有 code-simplifier 等少数需要深度理解的 agent 用 Opus。

5. **Rule 是否具体？** "写好代码"是废话。"函数不超过 50 行，文件不超过 800 行，嵌套不超过 4 层"是可执行的。ECC 的 AGENTS.md 中每一个编码规范都有具体数字。

## 模式 9：Plan-Orchestrate——多步骤计划的 agent 链编排

### 问题

你有一个 10 步的实施计划。每一步需要不同的 agent 参与——有的需要 planner，有的需要 code-reviewer，有的需要 security-reviewer。手动一个个调用很烦，但不手动又不知道怎么编排。

### ECC 的方案

ECC 的 `plan-orchestrate` skill 把计划文档自动翻译成 agent 链：

1. 读取计划文档，分解为步骤
2. 为每一步设计一个 agent 链（哪些 agent 按什么顺序参与）
3. 为每一步生成一个可直接粘贴的 `/orchestrate custom` 命令

例如，一个三步计划可能生成：

```
Step 1: /orchestrate custom "planner,architect" "设计用户认证模块的数据模型和 API 接口"
Step 2: /orchestrate custom "code-reviewer,security-reviewer" "审查 Step 1 的设计方案"
Step 3: /orchestrate custom "tdd-guide,code-reviewer" "按 Step 1 的设计实现，先写测试"
```

关键设计：**plan-orchestrate 只生成命令，不执行**。用户粘贴每一行时才执行。这保留了人类的控制权。

### 偷到自己工作流中

核心原则：**复杂计划应该预先编排 agent 链，而不是临场决定**。

当你有一个多步骤计划时，花 5 分钟写下每一步需要哪些 agent：

```markdown
## 实施计划

### Step 1: 需求分析
Agent: planner
输入: PRD 文档
输出: 分阶段的实施计划

### Step 2: 架构设计  
Agent: architect
输入: Step 1 的实施计划
输出: 数据模型 + API 设计

### Step 3: 安全审查
Agent: security-reviewer
输入: Step 2 的架构设计
输出: 安全问题清单

### Step 4: TDD 实现
Agent: tdd-guide
输入: Step 2 的架构设计 + Step 3 的安全修复
输出: 测试 + 实现代码
```

这比临场决定"现在我该用哪个 agent"效率高得多——因为你是在冷静的状态下做的编排决策，而不是在写到 Step 3 时临时想。

## 哪些设计不值得偷

公平起见，ECC 也有不值得照搬的设计：

**1. 249 个 skill 的规模本身。** ECC 的 WORKING-CONTEXT.md 承认部分 skill 存在"generic LLM rhetoric, canned CTA patterns, and forced platform stereotypes"，正在逐一审计。你不需要 249 个 skill——你需要 20 个写得好的。质量 > 数量。

**2. 79 个 command 的并存。** ECC 正在从 command 迁移到 skill，但目前两者并存。这是历史遗留，不是设计优势。新项目应该直接用 skill。

**3. 行业专项 skill。** ECC 有 healthcare-cdss-patterns、customs-trade-compliance、carrier-relationship-management 等行业 skill。除非你恰好在做这些行业，否则这些 skill 对你没有价值——它们是 ECC 追求"全面覆盖"的产物。

**4. ECC 2.0 的 Rust 后端。** `ecc2/` 目录是一个 Rust 编写的控制面，目前是 `2.0.0-rc.1`（alpha）。这是 ECC 的未来方向，但目前不成熟，不适合参考。

## 从 ECC 提取的 9 条设计原则

把上面的 9 个模式提炼成可记忆的设计原则：

1. **事实强制 > 自我评估。** 不要问 agent "你做对了吗？"——强制它去调查具体事实。（GateGuard）

2. **上下文是有限资源。** 像管理内存一样管理上下文窗口——审计、分级、定期回收。（Context Budget）

3. **对抗性思考需要隔离。** 想得到真正的第二意见，subagent 不能看到当前对话。（Council）

4. **自主循环必须有外部进度检测。** agent 不能判断自己是否在打转——用 commit hash 或测试结果从外部检测。（Loop Safety）

5. **改 prompt 之前先跑 eval。** 没有评估的工作流改进是猜测。（Eval-Driven Development）

6. **错误是学习材料。** 每次 agent 犯错，写一条 rule 防止再犯。知识积累靠的是持续的 lesson 记录。（Continuous Learning）

7. **分层防御。** 快速正则在 hook，深度分析在 skill，最后闸门在 commit。不同速度、不同深度的检查互补。（Hook Layering）

8. **改环境比改 prompt 更有效。** 上下文浪费、规则冲突、模型选择错误——这些环境因素对输出质量的影响比 prompt 措辞更大。（Harness Optimization）

9. **复杂计划预先编排。** 在冷静状态下设计 agent 链，而不是在实施过程中临时决定。（Plan-Orchestrate）
