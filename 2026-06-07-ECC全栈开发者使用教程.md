# ECC 全栈开发者使用教程：249 个 Skills + 63 个 Agents + 79 个 Commands 如何改变你的日常工作

ECC（Everything Claude Code）是目前最大的 AI 编码助手增强项目。182K+ stars，28K+ forks，170+ contributors，10+ 个月的密集日常使用积累。它不是一个工具——它是一个完整的"AI 编码工作流操作系统"。

仓库：[affaan-m/ECC](https://github.com/affaan-m/ECC) ⭐182K+ 🔀28K+ 📄MIT 📅2026-06-07

但对一个每天写代码的全栈开发者来说，这些数字没有意义。有意义的问题是：**装了这个东西之后，我的日常开发流程会怎么变？**

这篇文章回答这个问题。不讲架构、不讲设计哲学，只讲一个前后端全栈开发者从早到晚写代码时，ECC 的哪些部分在什么场景下能帮上忙。

---

## 对开发者有什么用

装完 ECC 之后，Claude Code（或其他支持的 agent harness）从一个"你问它答"的助手，变成了一个有专业分工的团队。具体来说：

- **写 TypeScript/React 前端**：输入 `/react-review`，一个专门的 react-reviewer agent 审查你的组件，检查 prop 类型、hooks 依赖、性能问题。写完后跑 `/react-test`，自动按 TDD 流程补测试。
- **写 Python/Django 后端**：写 Django ORM 查询时，django-security skill 自动注入 N+1 查询防护和 SQL 注入最佳实践。写完跑 `/python-review`，python-reviewer agent 检查类型标注、异步模式、错误处理。
- **构建时出错了**：不需要把错误信息粘给 Claude 解释——跑 `/react-build` 或 `/django-build`，专门的 build-resolver agent 自动分析错误栈，定位问题，尝试修复。
- **提交代码前**：`/security-scan` 跑一轮安全检查，`/code-review` 跑一轮质量审查，`/test-coverage` 确认覆盖率 ≥80%。三道闸门，全自动化。
- **接手一个陌生项目**：`/codebase-onboarding` 生成一份完整的代码库导览，包括架构决策、依赖关系、关键模块。新人 onboarding 时间从几天缩短到几小时。

这不是幻想场景。这是 ECC 作者 Affaan 在 10+ 个月的实际产品开发中打磨出来的工作流。

## 安装：5 分钟搞定

ECC 的安装是一个 Node.js 脚本，自动处理符号链接和配置：

```bash
# 克隆仓库
git clone https://github.com/affaan-m/ECC.git
cd ECC

# 运行安装脚本
./install.sh
```

安装脚本做了这几件事：

1. 将 `skills/` 目录下的 249 个 skill 链接到 `~/.claude/skills/`
2. 将 `commands/` 目录下的 79 个 command 链接到 `~/.claude/commands/`
3. 将 `agents/` 目录下的 63 个 agent 链接到 `~/.claude/agents/`
4. 将 `rules/` 目录下的编码规范链接到 `~/.claude/rules/`
5. 将 `hooks/hooks.json` 合并到用户级 hook 配置

装完之后，打开 Claude Code 就能看到所有新增的 `/` 命令和自动加载的 skill。

### 跨 Harness 支持

ECC 不只支持 Claude Code。它同时支持 Codex、Cursor、OpenCode、Gemini、Zed、GitHub Copilot。仓库中有专门的配置目录：

```
.cursor/hooks.json    # Cursor 适配
.gemini/GEMINI.md     # Gemini 适配
.opencode/            # OpenCode 适配
.zed/settings.json    # Zed 适配
.trae/                # Trae 适配
```

如果你同时使用多个 agent harness，ECC 的 skills 和 rules 可以跨工具共享。

## 日常开发中的 ECC：按场景拆解

### 场景 1：开始一个新功能

你接到一个任务："给用户模块加一个邮箱验证功能"。

**第一步：计划。** 输入 `/plan`，ECC 的 planner agent 介入。它不会直接开始写代码——它会：

- 分析代码库中现有的用户模块结构
- 识别需要修改的文件和新增的文件
- 列出依赖关系和风险点
- 生成一个分阶段的实施计划

planner agent 的设计哲学是：**复杂功能不规划就开写，返工成本是规划成本的 10 倍**。

**第二步：TDD。** 输入 `/tdd`，tdd-guide agent 接管。它强制执行 RED-GREEN-REFACTOR 循环：

1. **RED**：先写一个失败的测试。`describe('email verification', () => { it('sends verification email') ...`
2. **GREEN**：写最小实现让测试通过。
3. **REFACTOR**：重构，确认覆盖率 ≥80%。

这不是建议——ECC 的 AGENTS.md 明确写了"Minimum coverage: 80%"和"TDD workflow (mandatory)"。

**第三步：实现。** 按计划写代码。这个过程中，ECC 的 skills 会自动按需加载：

- 写 Django 视图时 → `django-patterns` skill 自动注入 DRF 序列化器、分页、过滤的最佳实践
- 写 React 组件时 → `react-patterns` skill 自动注入 hooks 模式、性能优化（useMemo/useCallback）规范
- 写数据库迁移时 → `database-migrations` skill 自动注入版本控制和回滚策略

**第四步：审查。** 写完代码后：

```
/code-review          # code-reviewer agent 检查代码质量
/react-review         # react-reviewer agent 专门审查前端
/python-review        # python-reviewer agent 专门审查后端
/security-scan        # security-reviewer agent 检查安全漏洞
```

四个 agent 可以并行跑，各自输出独立报告。code-reviewer 关注代码风格、函数大小、错误处理；security-reviewer 关注 SQL 注入、XSS、CSRF、硬编码密钥。

### 场景 2：构建失败了

npm run build 报了一堆 TypeScript 类型错误，或者 pytest 跑完有 import 失败。

**不要手动分析错误栈。** ECC 有专门的 build-resolver agent 家族：

| 命令 | Agent | 擅长 |
|------|-------|------|
| `/react-build` | react-build-resolver | TypeScript/webpack/vite 构建错误 |
| `/go-build` | go-build-resolver | Go 编译错误、依赖问题 |
| `/rust-build` | rust-build-resolver | Rust 编译错误、borrow checker |
| `/kotlin-build` | kotlin-build-resolver | Kotlin/Gradle 构建失败 |
| `/java-build` | java-build-resolver | Maven/Gradle 构建错误 |
| `/cpp-build` | cpp-build-resolver | C/C++ 编译链接错误 |
| `/flutter-build` | dart-build-resolver | Flutter/Dart 构建问题 |
| `/django-build` | django-build-resolver | Django 启动、依赖、迁移、collectstatic 失败 |
| `/gradle-build` | java-build-resolver | Gradle 特定的构建配置问题 |

每个 build-resolver agent 都经过专门训练来处理对应生态的常见构建错误模式。它不是通用 agent 拿到错误信息硬猜——它知道 `"Module not found: Can't resolve"` 在 webpack 5 中通常意味着什么，知道 `"error[E0382]"` 在 Rust 中是借用检查器报的什么错。

### 场景 3：代码审查 PR

你的同事提了一个 PR，你需要审查。或者你提了 PR，想在合并前自动审查一遍。

`/review-pr` 触发 pr-review-toolkit 中的六个专项 agent 并行审查：

| Agent | 查什么 | 典型发现 |
|-------|--------|----------|
| `comment-analyzer` | 注释是否和代码一致 | 注释说"返回 User"但实际返回 `User \| null` |
| `pr-test-analyzer` | 新代码是否有测试覆盖 | 新增的分支路径没有对应测试用例 |
| `silent-failure-hunter` | 静默失败 | `try { ... } catch {}` 空 catch 块 |
| `type-design-analyzer` | 类型设计是否合理 | `string` 用于有限状态值，应该用 union type |
| `code-simplifier` | 不必要的复杂度 | 三层嵌套的 if-else 可以简化为一个 guard clause |
| `code-reviewer` | 综合质量 | 违反项目 CLAUDE.md 中定义的编码规范 |

和传统 lint 工具（ESLint、Pylint）的区别：lint 只检查格式和常见反模式，ECC 的 agent 检查**语义层面的问题**——注释是否准确、测试覆盖是否充分、类型设计是否合理。这些是传统 lint 做不到的。

### 场景 4：接手一个陌生项目

你刚加入一个团队，或者接手一个前任留下的项目。代码库有 500 个文件，你完全不知道从哪里看起。

`/codebase-onboarding` skill 生成一份结构化的项目导览：

- 项目技术栈识别（语言、框架、构建工具、数据库）
- 核心模块划分和依赖关系图
- 关键设计决策的记录（从 git log 和注释中提取）
- 常用命令汇总（构建、测试、部署、数据库迁移）
- "从哪里开始读代码"的推荐阅读顺序

`/repo-scan` 做更深度的扫描——遍历所有源文件，生成代码库的"地图"，标注每个模块的职责和接口。

### 场景 5：写测试

你写了一堆代码，测试覆盖率不够。

`/test-coverage` 分析当前覆盖率报告，识别未覆盖的代码路径，为每个未覆盖的函数/分支生成测试用例。

`/e2e` 触发 e2e-runner agent，基于 Playwright 编写端到端测试。它不只是生成测试代码——它会：

1. 分析应用的路由和用户流程
2. 识别关键用户路径（登录 → 创建 → 提交 → 确认）
3. 为每条路径生成 Playwright 测试脚本
4. 运行测试，修复失败的用例

`/react-test`、`/rust-test`、`/kotlin-test`、`/go-test`、`/cpp-test`、`/fsharp-test` 分别是各语言生态的专项测试生成命令。

### 场景 6：提交和部署

代码写完了，测试通过了，审查过了。

`/pr` 命令帮你完成最后一步：

1. 分析所有 commit 的差异
2. 生成结构化的 PR 描述（变更概述、测试计划、影响范围）
3. 推送到远程并创建 PR

`/quality-gate` 是提交前的最终检查——它汇总 code-review、security-scan、test-coverage 的结果，给出一个通过/不通过的判断。只有全部通过，才建议你提交。

## Hooks：不叫就来的自动化

ECC 的 hook 系统是你不需要主动调用、但一直在背后运行的自动化。

### hooks.json 中的自动化

ECC 的 `hooks/hooks.json` 定义了在特定时机自动触发的行为：

**PreToolUse（工具调用前）**：

- 当 agent 调用 `Edit`/`Write` 时，自动检查是否涉及敏感文件（`.env`、密钥文件）
- 当 agent 执行 `Bash` 中的包管理器命令时，提醒使用 tmux 保持会话

**PostToolUse（工具调用后）**：

- 当 agent 修改了代码文件后，自动运行格式化和 lint
- 当 agent 运行了测试后，自动分析测试结果

**Stop（agent 完成响应时）**：

- 自动保存会话上下文
- 检查是否有未提交的变更，提醒创建 checkpoint

### 记忆持久化

`hooks/memory-persistence/` 目录包含一个 hook，在每次会话结束时自动将关键上下文（架构决策、调试发现、TODO）持久化到本地文件，下次会话恢复。

这解决了一个常见痛点：Claude Code 的上下文窗口有 200K token 限制，超过后会自动 compact（压缩），压缩过程中可能丢失重要上下文。记忆持久化 hook 确保关键信息不会因为 compact 而丢失。

```
csharp/      C# 规范
swift/       Swift 规范
fsharp/      F# 规范
dart/        Dart/Flutter 规范
angular/     Angular 规范
arkts/       ArkTS（HarmonyOS）规范
php/         PHP 规范
ruby/        Ruby 规范
perl/        Perl 规范
web/         Web 通用规范（HTML/CSS）
zh/          中文文档写作规范
```

这些规范在 Claude Code 启动时自动加载。当你写 TypeScript 代码时，TypeScript 的规范会自动生效——不需要手动引用，不需要在 prompt 中提醒。

通用规范（`common/`）中的核心原则：

- **不可变性**：永远创建新对象，不要修改已有对象
- **文件组织**：多个小文件优于少数大文件，200-400 行典型，800 行上限
- **错误处理**：每个层级都处理错误，UI 代码给用户友好提示，服务端记录详细上下文
- **输入验证**：在系统边界验证所有用户输入，使用 schema 验证，快速失败

## 已知问题

1. **上下文窗口膨胀**：ECC 的 249 个 skill 如果全部自动加载，会显著消耗上下文窗口。The Shortform Guide 明确警告："Your 200k context window before compacting might only be 70k with too many tools enabled." 建议保持 80 个以下活跃工具。

2. **Skill 质量参差**：WORKING-CONTEXT.md 中承认部分 skill 存在"generic LLM rhetoric, canned CTA patterns, and forced platform stereotypes"，正在进行逐一审计。

3. **Commands vs Skills 迁移**：79 个 command 中部分是 skill 的遗留 shim，部分是独立功能。官方正在将 command 迁移到 skill，但迁移完成前两者并存，可能造成混淆。

4. **ECC 2.0 仍是 alpha**：agent.yaml 和 Rust 编写的 ecc2/ 目录代表下一代架构，但标注为 `2.0.0-rc.1`，不建议在生产中使用。

5. **安装脚本依赖 Node.js**：即使你只用 Python/Go/Rust，安装 ECC 仍然需要 Node.js 环境。
