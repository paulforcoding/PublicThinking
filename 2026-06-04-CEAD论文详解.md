# CEAD 论文详解：Capability-Aligned Enterprise Agent Design

> **论文：** Designing Intelligent Enterprise Agents: A Capability-Aligned Multi-Agent Architecture  
> **作者：** John deVadoss（InterWork Alliance, Washington DC）  
> **发表：** arXiv:2605.08258，2026年5月7日  
> **领域：** cs.MA（Multi-Agent Systems）  
> **本地副本：** `/tmp/cead-2605.08258v1.pdf`

## 一、论文的核心论点

一句话总结：

> **治理（governance）是必要的，但不能成为组织 Agent 的首要抽象。首要抽象必须是 Agent 设计——能力边界、自主权分配、交互协议、工具和数据权限、状态和记忆设计、验证设计、人机交互设计。**

论文用了一整节来论证：SOA（面向服务架构）提供了契约、注册中心、策略、松耦合等好东西，但 Agent 和 Service 有本质区别——**Service 是被动等待调用的能力，Agent 是主动追求目标的行动者**。一个可以被功能正确性衡量，另一个必须额外衡量计划质量、工具选择、幻觉抵抗、提示注入抵抗、记忆安全性、委托安全性。

## 二、企业智能 Agent 的 7 个必要条件

论文先划了边界——不是所有带 LLM 的东西都叫 Agent。一个「企业智能 Agent」必须同时满足：

1. **目标驱动的意图处理**：接受的是目标（objective），不是过程调用
2. **LLM 介导的认知**：用 LLM 做解释、规划、推理、批判、工具路由
3. **工具介导的行动**：能调用企业服务、API、数据库、搜索、代码解释器等
4. **有界自主权**：在明确的权限、策略、审批边界内自主选择行动
5. **状态与记忆**：任务状态 + 短期上下文 + 可选长期记忆（作为企业数据治理）
6. **可观察的问责**：有所有者、身份、追踪、评估记录、成本记录、升级路径
7. **不确定性处理**：能检测歧义、请求澄清、寻求验证、或升级

**明确排除：** 简单 API 服务、确定性工作流、无状态 chatbot、松散包装的 prompt。

## 三、自主权的 5 级分层（L0-L4）

这是论文里非常有实操价值的一个设计：

| 级别 | 名称 | 能力 | 关键约束 |
|------|------|------|---------|
| **L0** | 观察 | 检索、总结、分类、解释 | 不改变任何系统 |
| **L1** | 起草 | 产出建议或草稿 | 人类执行 |
| **L2** | 准备 | 准备结构化行动 | 人类批准后提交 |
| **L3** | 有界执行 | 执行可逆/低风险操作 | 策略和预算限制内 |
| **L4** | 高自主执行 | 执行高影响工作流 | 需显式风险接受+持续监控+回滚机制 |

**关键洞察：** 企业不应该用二元的「Agent 能用 / 不能用」思维，而应该按行动的风险、可逆性、置信度来分配不同的自主权级别。一个 Agent 可以对某些操作是 L3，对另一些操作是 L1。

## 四、SOA 是范本，微服务是警示

论文花了很大篇幅做这个对比，对企业架构背景的人特别有共鸣：

**SOA 值得借鉴的：**
- 契约、注册中心、策略、松耦合、可观测性、版本管理

**SOA 不够用的：**
- 服务是被动的，Agent 是主动的、概率性的、有状态的、模型介导的
- Agent 需要额外的评估维度：计划质量、幻觉抵抗、提示注入防御、记忆安全、委托安全、不确定性校准

**微服务作为警示（原文）：**

> *"Widespread misunderstanding produced distributed monoliths, excessive operational overhead, and service sprawl. Intelligent agents could repeat this pattern at higher risk: instead of too many services, enterprises may create too many prompt-wrapped micro-agents with overlapping authority, opaque memory, inconsistent tool scopes, and untestable emergent behavior."*

翻译：微服务曾经因为「过度分解」产生了分布式单体、运维复杂性爆炸、服务蔓延。Agent 领域可能重蹈覆辙——而且风险更高——变成大量 prompt 包装的微 Agent，权限重叠、记忆不透明、工具范围不一致、行为不可测试。

---

## 五、论文定义的「微 Agent 泛滥」反模式

论文用了一整节描述这个最可能发生的企业反模式：

> *"The most likely enterprise anti-pattern is micro-agent proliferation: creating many small agents because microservices became familiar, because framework demos use role-playing agents, or because prompt wrappers appear cheap."*

这种模式会产生：
- 能力重复、所有权不明确
- 不必要的交接和长延迟链
- 不一致的工具权限
- 多个不受治理的记忆存储
- 评估覆盖差（每个 agent 只有局部测试，但系统有全局涌现行为）
- **级联错误**：一个 agent 的自信错误变成另一个 agent 的输入

**设计启示（原文）：**
> *"Start with business capabilities and risk boundaries, not with a target number of agents. Decompose only where there is a durable reason."*

## 六、CEAD 架构：5 层平面

```
意图与结果层 （Intent and Outcome）
    ↓
Agent 设计平面 （Agent Design Plane） ← 这是架构的核心
    ↓
Agent 运行时平面 （Agent Runtime Plane）
    ↓
企业能力平面 （Enterprise Capability Plane）
    ↓
支持控制与保障平面 （Supporting Control and Assurance）
```

**Agent 设计平面**是 CEAD 的灵魂——它不关心运行时用什么框架（LangGraph、AutoGen 还是自研），而是关心：
- 能力地图：哪些业务能力需要 Agent？
- Agent 边界：这个 Agent 拥有什么？不拥有什么？
- ACC（Agent Capability Contract）：每个 Agent 的设计契约
- 自主权分配：哪些操作 L0？哪些 L3？
- 交互拓扑：监督式？中介式？对等式？
- 验证策略：用什么来检验输出质量？
- 评估预言机：怎么定义「成功」？
- 退役规则：什么时候该下线一个 Agent？

## 七、Agent Capability Contract（ACC）——核心设计工件

这是 CEAD 最重要的发明，类似于 SOA 里的服务契约，但为 Agent 做了扩展：

| 契约字段 | 设计目的 |
|---------|---------|
| **业务能力与所有者** | 命名持久能力、问责产品负责人、运营团队、资金模式、退役负责人 |
| **目的与非目的** | 明确 Agent 被允许做什么，明确禁止做什么 |
| **自主权级别** | L0-L4 行动权限、审批要求、可逆性期望、回退行为 |
| **交互拓扑** | 监督式/专家/对等/验证者/中介者/确定性工作流辅助/人类辅助 |
| **输入输出模式** | 类型化的输入、输出、结构化计划、证据、引用、决策、错误格式 |
| **工具清单与范围** | 允许的工具、服务账号、OAuth 范围、速率限制、写入权限、禁止的工具 |
| **数据分类** | 允许的数据类别、保留期、驻留、隐私、跨租户限制 |
| **状态与记忆设计** | 短期上下文、长期记忆、写入权限、检索过滤器、中毒防御、删除、审计 |
| **模型行为策略** | 允许的模型家族、路由规则、上下文限制、温度上限、回退模型、接地要求 |
| **验证设计** | 确定性验证器、验证/批评提示、测试预言机、置信度阈值、矛盾检查、拒绝标准 |
| **人类交互** | 歧义、风险、成本、置信度、法律、财务、客户影响触发人类审批的条件 |
| **评估证据** | 离线测试、对抗性测试、回归套件、生产 Canary、验收阈值 |
| **可观测性与审计** | 追踪模式、跨度、模型/工具调用、token/成本指标、用户可见解释、事件回放 |
| **版本与废弃** | 控制 prompt、模型、工具、记忆、策略版本，带回滚和退役标准 |

> *"The ACC is not introduced as a bureaucratic form; it is the record of the design decision that justifies an agent's existence."*

ACC 不是官僚表格，而是一个 Agent 为什么应该存在的设计决策记录。

## 八、5 种运行时模式（从简到繁）

1. **监督式工具使用 Agent**（默认起点）：单一 Agent 规划并调用工具，一个 Supervisor 执行 ACC、验证输出、处理升级。**论文建议这是默认起点，因为它最小化协调风险。**

2. **中介式专家 Agent**：Supervisor 将任务委托给专家 Agent，仅在专家映射到独立的能力、数据边界、评估预言机或发布所有者时使用。

3. **验证者/挑战者 Agent**：仅当独立批判能实质性地提升安全性或正确性时才合理。不是泛泛的「第二意见」，而是有独立的预言机、证据源或策略目标。

4. **对等 Agent 协作**：A2A 类协议适用于独立拥有的 Agent 之间协作。但协议只提供通信，企业设计仍需定义身份、信任、契约、数据共享和升级路径。

5. **人在回路中的执行**：人类审批不是异常路径——对于不可逆、受监管、模糊或高影响的操作，它是交互设计的一部分。

## 九、12 条设计原则

1. **能力先于 Agent**：先定义业务能力和所有者，再定义 Agent 角色或 prompt
2. **边界先于拓扑**：先决定所有权、数据、行动、状态、评估边界，再决定用几个 Agent
3. **契约优先，非 prompt 优先**：每个生产 Agent 必须有 ACC
4. **少量、持久的 Agent**：偏好少量可问责 Agent，而非大量角色扮演 prompt 包装
5. **自主权是设计变量**：按行动风险、可逆性、置信度分配 L0-L4，不按模型能力
6. **最小权限遵循设计**：工具访问按任务、数据类别、行动类型、自主权级别限定
7. **记忆是被设计的状态**：需要所有权、保留期、删除、中毒防御、来源追踪、审查
8. **确定性检查先于概率性批判**：schema 验证、策略检查、业务规则验证在 LLM 批判之前
9. **评估系统，而非孤立 prompt**：衡量端到端任务成功、安全成功、工具误用、延迟、成本、可审计性
10. **协议是互操作通道，不是信任边界**：MCP、A2A 需要企业身份、授权、审计控制
11. **设计优雅降级**：Agent 必须知道何时停止、询问、升级或回退到确定性工作流
12. **退役 Agent**：未使用、低质量、重复或不安全的 Agent 应该像服务一样被废弃

## 十、实验：10,000 个企业任务的大规模对比

论文构建了 10,000 个企业任务，涵盖财务、HR、采购、IT、法务、销售、客服 7 个领域，每个任务标注了复杂度（1-5）、风险（1-5）、歧义度、敏感性、是否受监管、是否含对抗内容、所需工具数量、依赖关系。

### 五种架构对比：

| 架构 | 功能成功% | **安全成功%** | 自动安全% | 违规/千任务 | 记忆中毒/千任务 | 升级% | 审计覆盖% | 平均交接次数 | 平均成本 |
|------|----------|-------------|----------|-----------|---------------|------|----------|------------|---------|
| **A0** 提示工程单 Agent | 52.4 | 45.2 | 40.4 | 118.0 | 34.0 | 10.9 | 32.5 | 1.01 | 2.87 |
| **A1** 角色模拟微 Agent 蜂群 | 30.8 | **23.1** | 21.8 | 248.6 | 52.4 | 5.6 | 14.8 | 6.25 | 4.38 |
| **A2** SOA 中介 Agent | 63.1 | 58.8 | 44.2 | 54.0 | 22.5 | 25.7 | 68.1 | 2.37 | 3.51 |
| **A3** 治理优先但设计贫乏 | 53.2 | 50.8 | 30.7 | 30.0 | 18.5 | 40.9 | 80.6 | 4.51 | 4.55 |
| **A4** CEAD | **73.0** | **70.6** | 43.8 | **22.4** | **12.6** | 38.9 | **89.0** | 2.01 | 3.81 |

### 高风险/受监管/对抗性子集：

| 架构 | 安全成功% | 违规/千任务 | 记忆中毒/千任务 | 升级% | 审计% |
|------|----------|-----------|---------------|------|------|
| A0 提示工程 | 40.4 | 155.1 | 40.8 | 12.9 | 32.2 |
| A1 角色蜂群 | 19.4 | 313.7 | 64.2 | 6.4 | 14.5 |
| A2 SOA 中介 | 55.0 | 71.0 | 25.9 | 31.0 | 68.3 |
| A3 治理优先 | 47.8 | 38.2 | 23.4 | 48.3 | 80.1 |
| A4 CEAD | **67.8** | **29.5** | **16.1** | 46.3 | **88.7** |

### 关键解读：

- **A1（角色蜂群）是最差的**——功能成功率只有 30.8%，安全成功率仅 23.1%，违规率是 CEAD 的 11 倍（248.6 vs 22.4）。这就是「管理者 + 打工虾」模式——论文证明它是最危险的企业反模式。
- **A3（治理优先但设计贫乏）的教训最深刻**——它有最强的治理控制（高审计覆盖 80.6%、低违规率 30.0），但因为 Agent 太多（24 个）、能力边界模糊、交接成本高，安全成功率只有 50.8%，远不如 CEAD。**这说明治理不能替代好设计。**
- **CEAD 不是追求全自动化的**——它的「自动安全成功」仅 43.8%，比 SOA 中介（44.2%）还略低。这不是 bug，是 feature：CEAD 在 38.9% 的高风险任务中主动升级给人类。在受监管的企业场景中，**目标是安全完成，不是不惜代价的自动完成。**

## 十一、消融实验：每个设计要素贡献了什么？

| 移除的要素 | 安全成功% | 违规率变化 | 关键影响 |
|-----------|----------|----------|---------|
| CEAD 完整 | 70.1 | 23.9 | 基线 |
| **- 能力地图与 ACC** | 63.6 | 22.5 | 审计覆盖从 88.8% 暴跌到 71.7% |
| **- 运行时策略引擎** | 66.9 | **44.9** → 违规几乎翻倍 | |
| **- 验证/批评门控** | 63.9 | 29.2 | 功能和安全成功率都显著下降 |
| **- 最小权限工具范围** | 68.9 | **43.0** → 违规几乎翻倍 | |
| **- 记忆治理** | 65.5 | 20.8 | 记忆中毒从 14.4 飙升到 **28.5** |
| **- 人类审批门控** | 70.8 | 24.1 | 升级率从 39.4% 暴跌到 4.1%，审计下降 |

**最重要的发现：** 能力地图 + ACC 和验证门控对**功能质量**影响最大；策略引擎和最小权限对**安全合规**影响最大；记忆治理直接影响数据安全。没有任何一个要素是可以省略的。

## 十二、Agent 数量与可靠性的关系——「越多越糟」的量化证明

论文做了一个「Agent 数量扫描」（1 到 64 个）：

- **无治理系统中**：Agent 超过 16-32 个后，安全成功率急剧下降——因为交接成本、攻击面、协调失败压倒了并行收益
- **CEAD 中**：趋势更平稳，但在 64 个 Agent 时也开始退化

**结论：** 使用能够表达不同能力、风险边界、状态边界、评估边界和所有权边界所需的最少 Agent 数量。

## 十三、什么时候该用多 Agent？什么时候不该？

**多 Agent 设计是合理的，当且仅当至少满足以下条件之一：**

- 不同业务能力有不同的问责所有者
- 不同 Agent 需要实质上不同的数据或工具权限
- 任务需要不同的评估预言机
- 任务需要独立的发布节奏或供应商隔离
- 一个 Agent 必须验证、挑战或批准另一个 Agent 的工作
- 需要与外部或独立治理的 Agent 协作

**不合理的情况：**

> *"It is not justified merely because multiple roles can be named in a prompt. Role names are not architecture."*

仅仅因为你能在 prompt 里写出多个角色名称，不意味着你应该创建多个 Agent。**角色名称不是架构。**

## 十四、13 条企业实施建议

1. 从能力地图开始——在命名 Agent 之前，识别业务能力、所有者、数据类别、行动和风险
2. 要求每个生产 Agent 有 ACC——没有目的、权限、工具、记忆、评估证据和所有者，就不能上线
3. 偏好监督者+专家模式，而非开放式蜂群——从单个监督式工具使用 Agent 开始；仅在边界有合理理由时才加专家
4. 将 SOA 和微服务资产当作工具使用，而非 Agent 的替代品——服务提供能力，Agent 决定如何在设计的权限边界内使用能力
5. 让治理支持设计，而非主导设计——身份、策略、审计、审查和事件响应应该执行设计、检测漂移、提供保障
6. 设计最小权限——按任务、数据类别、行动和自主权级别限定工具范围。一个有广泛写入权限的通用 Agent 是企业级安全隐患
7. 把记忆当作企业状态来设计——记忆需要保留期、访问控制、来源追踪、删除、中毒检测和审查
8. 将互操作性与信任分离——MCP 和 A2A 可以减少集成摩擦，但企业身份、授权、策略、审计和数据控制仍然是强制性的
9. 衡量安全成功，而非仅仅是完成——包括策略违规、记忆中毒、幻觉、工具误用、成本、延迟、审计覆盖和升级质量
10. 采用评估即发布门控——工具单元测试、黄金任务套件、对抗性测试、回归测试、离线评估、Canary 和生产监控应是强制性的
11. 限制 Agent 扩散——每季度审查重复的 Agent。合并共享所有者、数据、工具、评估和生命周期的 Agent
12. 让人类成为系统的一部分——人类审批应是显式的、可观察的、基于风险的，而非事后补救
13. 规划事件响应——企业需要针对 Agent 事件的 kill switch、回滚、记忆隔离、取证追踪和策略撤销

## 十五、对「管理者 Agent + 打工虾 Agent」场景的直接分析

部门经理的想法从 CEAD 框架下看：

| 他的想法 | CEAD 框架的诊断 |
|---------|---------------|
| 一个管理者 Agent | ✅ 合理——这是一个 **Supervisor/Orchestrator**，负责任务分解、调度和结果聚合 |
| N 个打工虾（按系统分） | ⚠️ 危险——如果每个系统一个 Agent，这就是微 Agent 蜂群（论文的 A1 模式，23.1% 成功率） |
| 管理者汇总后发给经理 | ✅ 合理——但缺少中间验证层 |

**正确的做法应该是：**

1. 不要把 Agent 映射到「人」（打工虾），而是映射到「业务能力」
2. 系统健康检查这件事，合理的分解可能是：
   - 一个 **数据采集能力**（统一接口采集各系统指标）
   - 一个 **异常检测能力**（分析数据、识别异常）
   - 一个 **报告生成能力**（格式化输出）
   - 一个 **Orchestrator** 负责编排流程
   - 一个 **Verifier** 做最终输出质检
3. 总共 4-5 个 Agent，而不是 N+1 个
4. 每个 Agent 有明确的 ACC（特别是数据分类、工具权限、自主权级别）
5. 高风险动作（如重启系统）应该是 L2（准备+人类审批），而不是自动化

## 附录：论文中的关键概念速查

- **安全成功（Safe Success）：** 功能成功 AND 无策略违规 AND 无记忆中毒
- **ACC（Agent Capability Contract）：** Agent 的设计契约，回答「为什么这个 Agent 应该存在」
- **微 Agent 泛滥（Micro-agent Proliferation）：** 企业最可能犯的反模式——创建过多角色扮演式 Agent
- **L0-L4 自主权：** 从只观察到高自主执行的五级分层，按风险分配，不按模型能力
- **设计平面（Design Plane）：** CEAD 的核心——独立于运行时框架的 Agent 设计决策层
