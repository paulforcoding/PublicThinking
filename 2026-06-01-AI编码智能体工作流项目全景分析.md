## 引言

2025-2026 年，AI 编码智能体（Coding Agent）迎来了爆发式增长。从 Claude Code 的插件生态到 OpenAI Codex CLI、Gemini CLI、Cursor、Zed 等多款 AI 编程工具，开发者不再满足于「单次问答」模式，而是开始追求**工作流自动化**、**多智能体编排**、**上下文工程**等更高阶的能力。

本报告对 GitHub 上最具代表性的 **7 个 AI 编码智能体工作流项目** 进行了深度技术分析。这些项目覆盖了从「大而全的超级平台」（ECC，63 agents + 249 skills）到「轻量级方法论集合」（agent-skills，23 skills；skills，18 skills），从「中文本地化增强」（superpowers-zh）到「前沿联邦架构实验」（ruflo）。

### 1.3 项目定位

## 二、核心亮点（5 个突出设计）

### 2.1 多 harner 统一适配层（Cross-Harness Architecture）

ECC 最独特的设计是它不绑定单一 AI 工具。通过 `.claude-plugin/`、`.codex/`、`.cursor/`、`.opencode/`、`.gemini/`、`.zed/`、`.codebuddy/`、`.trae/` 等多目录结构，同一套 agents/commands/skills/rules 可以部署到 7+ 个不同的 AI Agent Harness：

```
.claude-plugin/plugin.json    → Claude Code 插件市场
.codex/                        → OpenAI Codex CLI
.cursor/                       → Cursor IDE
.opencode/                     → OpenCode (含 plugins/tools，TypeScript 实现)
.gemini/                       → Google Gemini
.zed/                          → Zed Editor
.codebuddy/                    → CodeBuddy
.trae/                         → Trae IDE
.qwen/                         → Qwen CLI
```

具体证据来自 `scripts/lib/install-targets/` 目录：
```
install-targets/
  claude-home.js        cursor-project.js
  codex-home.js         codebuddy-project.js
  opencode-home.js      joycode-project.js
  gemini-project.js     qwen-home.js
  zed-project.js
```

每个 target 都有独立的安装适配器，自动将 ECC 的核心资产（agents、commands、skills、rules）转换成对应 harness 的原生格式。例如 OpenCode 支持 20+ 种事件类型的 plugin hooks，Codex 需要 `codex.md` 格式，而 Claude Code 使用 `plugin.json` + `marketplace.json`。

### 2.2 选择性安装架构（Selective Install）

ECC 实现了精密的**声明式安装系统**（v1.9.0 引入），支持按需安装特定组件，避免上下文窗口膨胀：

**三层清单结构**（`manifests/` 目录）：

```
install-profiles.json     → 7 种预定义安装策略
install-modules.json      → 23 个功能模块定义
install-components.json   → 细粒度组件到模块的映射
```

**7 种安装策略**：

| 策略 | 模块数 | 适用场景 | 上下文成本 |
|------|--------|----------|-----------|
| `minimal` | 5 | 低上下文/无 hook 场景 | 极轻 |
| `core` | 6 | 基础 harness 基线 | 轻 |
| `developer` | 9 | 日常工程开发 | 中 |
| `security` | 7 | 安全审查为主 | 中 |
| `research` | 9 | 研究/内容创作 | 中 |
| `full` | 23 | 完整能力 | 重 |
| 自定义 | 按需 | 组合 `--with`/`--without` 标志 | 可变 |

**精准咨询式安装**（`npx ecc consult`）：
```bash
npx ecc consult "security reviews" --target claude
# 返回匹配组件、推荐策略和安装预览
```

安装过程中的 `install-plan.js` 生成文件级安装计划，`install-apply.js` 执行并记录到 SQLite 状态存储，`doctor.js`/`repair.js` 提供诊断和自我修复能力。

### 2.3 多层 Hook 系统与运行时控制

ECC 的 Hook 系统是一个**生产级的工具生命周期守卫系统**，包含 6 大事件类型和约 20 个活跃 hook：

**Hook 生命周期全景**（来自 `hooks/hooks.json`）：

```
PreToolUse (8 hooks):
  ├── pre:bash:dispatcher        —— Bash 前检查（质量/tmux/push/GateGuard）
  ├── pre:write:doc-file-warning —— 非标准文档文件警告
  ├── pre:edit-write:suggest-compact —— 智能压缩建议（每~50次工具调用）
  ├── pre:observe:continuous-learning —— 工具使用观察（异步，用于持续学习）
  ├── pre:governance-capture     —— 治理事件捕获（密钥/策略违规）
  ├── pre:config-protection      —— 阻止修改 linter/formatter 配置
  ├── pre:mcp-health-check       —— MCP 服务器健康检查
  └── pre:edit-write:gateguard-fact-force —— 首次编辑文件前强制审查

PostToolUse (8 hooks):
  ├── post:bash:dispatcher       —— Bash 后日志/PR/构建通知（异步30s）
  ├── post:quality-gate          —— 编辑后质量门禁（异步30s）
  ├── post:edit:design-quality-check —— 前端 UI 风格漂移检测
  ├── post:edit:accumulator      —— 记录编辑文件，Stop 时批量格式化
  ├── post:edit:console-warn     —— console.log 警告
  ├── post:session-activity-tracker —— 会话活动追踪（面向 ECC2）
  ├── post:observe:continuous-learning —— 工具输出观察
  ├── post:ecc-metrics-bridge    —— 运行时会话指标
  └── post:ecc-context-monitor   —— 上下文耗尽/高成本/作用域蠕变警告

PostToolUseFailure (1 hook):
  └── post:mcp-health-check      —— MCP 故障跟踪和重连

Stop (6 hooks):
  ├── stop:format-typecheck      —— 批量格式化+类型检查（300s timeout）
  ├── stop:check-console-log     —— console.log 审计
  ├── stop:session-end           —— 持久化会话状态
  ├── stop:evaluate-session      —— 评估可提取模式（持续学习）
  ├── stop:cost-tracker          —— Token/成本追踪
  └── stop:desktop-notify        —— macOS 桌面通知

SessionStart (1 hook):
  └── session:start              —— 加载上下文、检测包管理器

PreCompact (1 hook):
  └── pre:compact                —— 压缩前保存状态

SessionEnd (1 hook):
  └── session:end:marker         —— 会话结束生命周期标记
```

**运行时控制亮点**：

- **Hook 配置分级**：`ECC_HOOK_PROFILE=minimal|standard|strict` 环境变量切换钩子严格程度
- **动态禁用**：`ECC_DISABLED_HOOKS="pre:bash:tmux-reminder,post:edit:typecheck"`
- **插件根目录自动发现**：每个 hook 命令都包含内联的 bootstrap 代码，自动扫描 6 种可能的安装路径
- **异步执行**：非阻塞 hook 使用 `"async": true` + `"timeout"` 配置

### 2.4 持续学习系统（Continuous Learning）

ECC 实现了从会话中自动提取模式和知识的持续学习系统：

**关键技术架构**（`scripts/lib/`）：

```
state-store/          —— SQLite 状态存储
  ├── schema.js       —— 数据库 schema（sessions, observations, skills）
  ├── queries.js      —— 查询接口
  ├── migrations.js   —— schema 迁移
  └── index.js        —— 状态存储入口

skill-evolution/      —— Skill 自我进化
  ├── versioning.js   —— 版本管理
  ├── tracker.js      —— 使用追踪
  ├── provenance.js   —— 来源追溯
  ├── health.js       —— 健康评分
  └── dashboard.js    —— 技能仪表盘

skill-improvement/    —— Skill 改进
  ├── observations.js —— 观察收集
  ├── health.js       —— 健康评估
  ├── evaluate.js     —— 效果评估
  └── amendify.js     —— 改进建议

session-adapters/     —— 多源会话适配
  ├── claude-history.js   —— Claude 历史
  ├── dmux-tmux.js        —— tmux dmux 日志
  └── canonical-session.js —— 规范化会话格式
```

**工作流程**：
1. `observer-sessions.js` + `observe-runner.js` hook 在每次工具调用前后捕获观察数据
2. `evaluate-session.js` Stop hook 在会话间歇评估是否有可提取的模式
3. `skill-evolution/` 系统自动追踪 skill 的使用频率、成功率、改进建议
4. 通过 `/learn` 和 `/learn-eval` 命令手动触发学习评估

### 3.3 扩展机制

ECC 的核心扩展机制基于以下设计模式：

**1. Agent 扩展**（`agents/*.md`）- YAML Frontmatter 定义：
```yaml
---
name: planner
description: Expert planning specialist...
tools: ["Read", "Grep", "Glob"]    # 工具权限范围
model: opus                         # 建议模型
---
# Prompt Defense Baseline
... (prompt 内容)
```

**2. Skill 扩展**（`skills/<name>/SKILL.md` + 辅助文件）：
- `SKILL.md` - 核心 prompt（含 frontmatter、激活条件、工作流程）
- `scripts/` - 辅助脚本
- `reference/` - 参考文档（如 videodb 的 API 文档）
- `references/` - 参考资料

**3. Rules 分层覆盖**（`rules/`）：
- `common/` 定义通用规则
- `<language>/` 定义语言特定规则（引用 common，特定覆盖通用）

**4. Hook 注册**（`hooks/hooks.json`）：
- 每个 hook 有唯一 ID、matcher 条件、command 执行体和 profile 控制

**5. 安装清单**（`manifests/`）：
- 三层 JSON 声明系统，支持 `--with`/`--without` 细粒度控制

## 六、技术细节

### 6.1 Hook 插件根目录自发现机制

ECC 的 hook 命令执行体（写入 `hooks.json` 的 inline JS）包含一个精巧的**自发现 bootstrap**。每个 hook 命令的前半部分是内联的路径解析器，按优先级扫描 6 种可能的安装位置：

```javascript
// 伪代码还原
function resolvePluginRoot() {
  if (process.env.CLAUDE_PLUGIN_ROOT) return it;

  // 1. 检查 ~/.claude/（手动安装场景）
  // 2. 检查 ~/.claude/plugins/ecc/
  // 3. 检查 ~/.claude/plugins/ecc@ecc/（marketplace 场景）
  // 4. 检查 ~/.claude/plugins/marketplaces/ecc/
  // 5. 检查 ~/.claude/plugins/everything-claude-code/（legacy slug）
  // 6. 扫描 ~/.claude/plugins/cache/ecc/ 找 semver 目录
  // 7. 扫描 ~/.claude/plugins/cache/everything-claude-code/
  // 8. fallback: ~/.claude/
}
```

这种设计使得同一个 `hooks.json` 可以跨多种安装方式工作，不需要用户手动配置路径。

### 6.2 SQLite 状态存储

ECC 使用 `sql.js`（浏览器端 SQLite WASM 实现）作为跨平台状态存储，schema 定义在 `scripts/lib/state-store/schema.js`：

**存储内容**：
- 安装状态（哪些组件已安装，版本号）
- 会话记录（开始时间、工具调用计数、成本估算）
- Skill 进化数据（使用频率、成功率、版本历史）
- 观察数据（用于持续学习）
- 工作项跟踪（Linear/GitHub/handoff）

**特点**：
- 零外部依赖（sql.js 纯 JS + WASM）
- 支持迁移（migrations.js）
- 可通过 `npx ecc status` 和 `npx ecc sessions` 查询

### 6.3 Skill 进化系统

`scripts/lib/skill-evolution/` 实现了 skill 的自我改进闭环：

```
观察收集 (observer-sessions.js)
  → 数据存储 (SQLite skill_observations 表)
    → 评估分析 (evaluate.js - 使用频率/成功率/改进建议)
      → 版本追踪 (versioning.js + tracker.js)
        → 来源追溯 (provenance.js)
          → 健康仪表盘 (dashboard.js)
            → 改进建议 (amendify.js)
```

这使 skill 可以：
- 追踪使用模式和成功率
- 自动识别哪些 skill 需要更新
- 通过 `/learn` 命令从会话中提取新 skill
- 版本化管理 skill 的演进

### 6.4 多 Harness 适配机制

不同 harness 有截然不同的配置格式，ECC 的适配策略：

| Harness | Agent 格式 | Command 格式 | Hook 格式 | 配置位置 |
|---------|-----------|-------------|----------|---------|
| Claude Code | `agents/*.md` (frontmatter) | `commands/*.md` | `hooks/hooks.json` | `~/.claude/` |
| Codex | `AGENTS.md` + `.codex/` | `.codex/commands/` | `.codex/hooks/` | `~/.codex/` |
| OpenCode | `.opencode/prompts/agents/*.txt` | `.opencode/commands/*.md` | TypeScript plugin | `.opencode/` |
| Cursor | `.cursor/rules/` | IDE 内置 | IDE 内置 | `.cursor/` |
| Gemini | 规则文件 | 规则文件 | 内置 | `.gemini/` |

安装时通过 `scripts/lib/install-targets/` 下的适配器自动转换。

### 6.5 Context 管理策略

ECC 对上下文窗口管理有精细控制：

**分层策略**：
1. **System Prompt 瘦身**：`rules/` 按需安装（不装不需要的语言规则）
2. **Selective Install**：通过 7 种策略控制安装体量
3. **Strategic Compact**：`pre:edit-write:suggest-compact` hook 在每 ~50 次工具调用后提醒压缩
4. **SessionStart 上下文控制**：`ECC_SESSION_START_MAX_CHARS` 限制加载的历史上下文量
5. **Context Monitor**：`post:ecc-context-monitor` hook 实时检测上下文耗尽、高成本、作用域蠕变

**环境变量微调**：
```bash
ECC_SESSION_START_MAX_CHARS=4000   # 限制 SessionStart 注入量
ECC_SESSION_START_CONTEXT=off      # 完全禁用 SessionStart 上下文
ECC_CONTEXT_MONITOR_COST_WARNINGS=off  # 抑制成本告警
```

### 6.6 安全防护设计

**AgentShield 集成**：独立 npm 包 `ecc-agentshield`，1282 个测试、102 条安全规则

**Prompt Defense Baseline**：每个 agent prompt 开头都有标准化防护指令，防止：
- 角色/身份篡改
- 机密数据泄露
- 代码注入（unicode homoglyphs、零宽字符等）
- 越狱攻击

**Config Protection Hook**：`pre:config-protection` 阻止 AI 修改 linter/formatter 配置，引导修复代码而非削弱检查

**GateGuard Fact-Force**：首次编辑/写入文件前强制审查（导入、数据 schema、用户指令）

**Governance Capture**：可选的事件捕获系统（通过 `ECC_GOVERNANCE_CAPTURE=1` 启用），记录密钥泄露、策略违规等

## 八、总结

ECC 是目前开源社区中规模最大、覆盖最全面的 AI coding agent 操作系统。它的核心价值不在于某一个功能，而在于**将 10+ 个月的真实工程经验系统化**为可安装、可配置、可进化的基础设施：

1. **规模惊人**：63 agents + 249 skills + 79 commands + 20 规则集 + ~2000 提交
2. **工程深度**：不只是 prompt 集合，而是完整的选择性安装系统、Hook 运行时、状态存储、持续学习闭环
3. **跨 harness**：同一套资产适配 7+ 个 AI 工具，降低 lock-in 风险
4. **生产导向**：安全防护、治理捕获、成本追踪、上下文监控都是面向生产级使用
5. **面向未来**：ECC 2.0 Rust 控制面已在 tree 中构建，为多会话编排和 operator 工作流做准备

从全栈开发者视角看，ECC 最值得学习的设计模式是：**通过声明式清单系统（manifests）+ 多 target 适配器（install-targets）+ 运行时 hook 注入，实现了一套"write once, run across harnesses"的 AI agent 基础设施**。这类似于 Docker 对容器运行时的抽象，ECC 在 AI agent 层面做了类似的事情。

**Get Shit Done**（以下简称 GSD）是一个**元提示（Meta-Prompting）、上下文工程（Context Engineering）与规约驱动开发（Spec-Driven Development）系统**，专为 AI 编程助手（Claude Code、Gemini CLI、Codex、OpenCode 等 14+ 运行时）设计。它解决的核心问题是 **Context Rot**：随着 AI 上下文窗口被填满，输出质量逐步劣化的工程难题。

## 二、核心亮点

### 2.1 上下文工程：解决 Context Rot 的系统性方案

GSD 最核心的创新在于**上下文工程**——不依赖单个巨型 prompt 完成所有工作，而是通过结构化的文件体系分段注入上下文。每个文件都有精确的大小阈值设计，基于 Claude 输出质量开始退化的实测数据。

以 `.planning/` 目录下的上下文文件体系为例：

```
.planning/
├── PROJECT.md          # 项目愿景（始终加载）
├── REQUIREMENTS.md     # 需求定义（带 ID 和阶段可追踪性）
├── ROADMAP.md          # 路线图（去哪、已完成什么）
├── STATE.md            # 决策、阻塞项、跨会话记忆
├── research/           # 生态知识（技术栈、功能、架构、坑点）
├── phases/XX-phase/
│   ├── CONTEXT.md      # 用户决策上下文
│   ├── RESEARCH.md     # 领域调研
│   ├── XX-YY-PLAN.md   # 原子执行计划（XML 格式）
│   ├── XX-YY-SUMMARY.md # 执行结果
│   └── VERIFICATION.md # 验证报告
├── spikes/             # 可行性实验
├── sketches/           # HTML 原型
├── threads/            # 跨会话知识线程
└── seeds/              # 前瞻性想法
```

**佐证代码**：`get-shit-done/bin/lib/state.cjs`（1,940 行）负责 `STATE.md` 的读写、进度计算、前后端耦合管理；`get-shit-done/bin/lib/prompt-budget.cjs`（12,297 字节）专门做上下文预算控制。`CONTEXT.md`（87,446 字节，相当于一本技术手册）详细记录了所有模块的职责边界和数据流向。

### 2.2 Wave 并行执行引擎

GSD 的核心执行模型是 **Wave-based Parallel Execution**：将阶段计划按依赖关系分组成"wave"，同一 wave 内的计划并行执行（每个计划一个全新的子代理上下文），不同 wave 之间顺序推进。

数据结构来源于 `commands/gsd/execute-phase.md`：

```
WAVE 1 (parallel)          WAVE 2 (parallel)          WAVE 3
┌─────────┐ ┌─────────┐    ┌─────────┐ ┌─────────┐    ┌─────────┐
│ Plan 01 │ │ Plan 02 │ →  │ Plan 03 │ │ Plan 04 │ →  │ Plan 05 │
└─────────┘ └─────────┘    └─────────┘ └─────────┘    └─────────┘
     │           │              ↑           ↑              ↑
     └───────────┴──────────────┴───────────┘              │
            Dependencies cross wave boundaries
```

**实现细节**：
- 每个执行代理获得**全新的 20 万 token 上下文**，无历史垃圾
- 每个任务完成后**原子 git commit**（Conventional Commits 格式）
- orchestrator 通过 `gsd-sdk query init.execute-phase` 获取阶段状态，分配计划到 wave，并行拉起 `gsd-executor` 子代理
- 主上下文窗口维持 30-40% 利用率，重活在子代理中进行

**佐证代码**：`agents/gsd-executor.md`（36,458 字节/774 行）定义了完整的执行流程（加载→验证→执行→提交→检查点→验证→摘要→状态更新，共 8 步）；`commands/gsd/execute-phase.md` 是 orchestrator 的命令契约。

### 2.3 双轨架构：CJS 核心 + TypeScript SDK 的"硬缝"

GSD 最值得全栈开发者关注的工程决策是其**双轨架构**：

| 层级 | 技术 | 职责 |
|------|------|------|
| **核心库（CJS）** | CommonJS `.cjs`，零外部依赖 | `gsd-tools` CLI 工具（~300 命令），状态管理、安装器 |
| **SDK** | TypeScript ESM/NodeNext | 程序化 API，查询注册表，Phase Runner |
| **同步桥** | Worker 线程 + `synckit`（`Atomics.wait`） | 将 SDK 异步操作桥接到同步 CJS 调用方 |
| **命令路由中枢** | `command-routing-hub.cjs` | 统一调度层，CJS/SDK 模式选择 |

**"硬缝"设计**的具体实现（来自 `CLAUDE.md` 和 `CONTEXT.md`）：

1. **共享模块**：纯逻辑抽取到 `.ts` 源文件，代码生成器生成 `.cjs` + `.ts` 副本（如 `configuration.generated.cjs`、`state-document.generated.cjs`、`secrets.generated.cjs` 等），新鲜度检查器（`lint-shared-module-handsync.cjs`）验证无漂移。

2. **同步运行时桥**：`sdk/src/runtime-bridge-sync/` 使用 `synckit`（基于 `SharedArrayBuffer` + `Atomics.wait`）在 Worker 线程中运行 SDK 异步查询，向 CJS 调用方暴露同步接口。首次调用 ~80ms（Worker 启动），稳态 ~0.1ms/次。这使 100+ CJS 消费者能调用 SDK 能力而无需子进程跳板。

3. **命令路由中枢**：`get-shit-done/bin/lib/command-routing-hub.cjs`（7,956 字节）定义了统一的 `hub.dispatch({ family, subcommand, args, cwd }) → Result` 接口，6 种错误类型闭集（`UnknownCommand`/`InvalidArgs`/`HandlerRefusal`/`HandlerFailure`/`SdkLoadFailed`/`SdkDispatchFailed`），永不抛异常、永不打印、永不退出。

**佐证**：`sdk/src/index.ts` 导出 `GSD` 类，提供 `executePlan()`、`executePhase()`、`run()` 等高级 API；`get-shit-done/bin/lib/cjs-sdk-bridge.cjs` 实现桥接。

### 2.4 XML 结构化任务计划格式

GSD 使用为 Claude 优化的 XML 格式定义原子任务，指令精确到不需要 AI 猜测：

```xml
<task type="auto">
  <name>Create login endpoint</name>
  <files>src/app/api/auth/login/route.ts</files>
  <action>
    Use jose for JWT (not jsonwebtoken - CommonJS issues).
    Validate credentials against users table.
    Return httpOnly cookie on success.
  </action>
  <verify>curl -X POST localhost:3000/api/auth/login returns 200 + Set-Cookie</verify>
  <done>Valid credentials return cookie, invalid return 401</done>
</task>
```

每个计划包含前端元数据（phase, plan, type, autonomous, wave, depends_on）、目标任务列表（带 `<verify>` 和 `<done>` 内建验证）、以及 `@~/.claude/get-shit-done/references/` 的系统引用。这种格式让 Planner（`agents/gsd-planner.md`，49,350 字节）生成的计划能被 Executor 零歧义执行。

**佐证**：`sdk/src/plan-parser.ts`（14,774 字节）实现了 XML 计划的解析器；`sdk/src/plan-parser.test.ts`（16,499 字节）有完整的解析测试覆盖。

### 2.5 多代理编排体系

GSD 有 33 个专用 AI Agent（`agents/gsd-*.md`），覆盖研发全生命周期：

| Agent | 文件大小 | 角色 |
|-------|---------|------|
| `gsd-planner` | 49,350 B | 阶段规划、任务分解、依赖分析 |
| `gsd-executor` | 36,458 B | 计划原子执行 |
| `gsd-verifier` | 35,730 B | 对照目标验证交付 |
| `gsd-code-reviewer` | 16,776 B | 跨 AI 同行评审 |
| `gsd-codebase-mapper` | 21,388 B | 现有代码库分析与映射 |
| `gsd-debugger` | 46,783 B | 系统化调试、根因分析 |
| `gsd-phase-researcher` | 37,266 B | 阶段领域研究 |
| `gsd-plan-checker` | 36,076 B | 计划质量验证 |
| `gsd-doc-writer` | 38,059 B | 文档自动生成 |
| `gsd-roadmapper` | 19,133 B | 路线图生成与管理 |

每个 Agent 定义都是自包含的 Markdown 文件（含 frontmatter 元数据），包含完整的角色指令、工具限制、上下文发现流程和执行步骤。这种设计使 Agent 可以独立被不同运行时（Claude、Codex、Gemini 等）加载。

## 四、功能体系

### 4.1 完整工作流设计

GSD 定义了一个 **6 阶段循环**：

```
 /gsd-new-project                    ← 创建新项目（提问→研究→需求→路线图）
      │
      ▼
 /gsd-discuss-phase N  ────────────── 讨论阶段 N 的实现偏好
      │                                  ↓ 生成 CONTEXT.md
      ▼
 /gsd-plan-phase N     ────────────── 研究 + 规划 + 验证
      │                                  ↓ 生成 RESEARCH.md + PLAN.md
      ▼
 /gsd-execute-phase N  ────────────── Wave 并行执行
      │                                  ↓ 生成 SUMMARY.md + VERIFICATION.md
      ▼
 /gsd-verify-work N    ────────────── 人工 UAT 验证
      │                                  ↓ 生成 UAT.md / 修复计划
      ▼
 /gsd-ship N           ────────────── 创建 PR，代码发布
      │
      ▼
 /gsd-complete-milestone ──────────── 归档里程碑，打 release tag
      │
      ▼
 /gsd-new-milestone    ────────────── 开始下一个版本
```

**快速模式**：`/gsd-quick [--discuss] [--research] [--full]` 保留原子提交和状态跟踪，但跳过可选步骤，适合临时任务。

**自主模式**：配置 `workflow.auto_advance=true` 可自动串联 discuss → plan → execute。

### 4.2 提示词策略

GSD 的提示词策略有三个层次：

**层次一：系统级元提示**
- 33 个 Agent 定义文件，每个都是自包含的角色指令
- 安装器为不同运行时做格式转换（Claude Code 用 skill 格式，Codex 用 `SKILL.md`，Cline 用 `.clinerules`）
- `--minimal` 模式将系统提示开销从 ~12K token 降至 ~700 token

**层次二：阶段上下文**
- `PROJECT.md`：项目愿景（始终加载）
- `CONTEXT.md`：用户决策（讨论阶段产物，喂给后续 Planner 和 Researcher）
- `RESEARCH.md`：领域知识（规划阶段产物）
- `STATE.md`：跨会话记忆（决策、阻塞项、当前位置）

**层次三：原子任务级**
- XML 格式计划包含精确指令 + 内建验证
- 每个 executor 子代理获得全新 20 万 token 上下文
- `commands/gsd/plan-phase.md` 定义了 orchestrator 如何组装提示词

### 4.3 上下文管理策略

GSD 的上下文管理核心是**信息分层 + 按需加载**：

| 机制 | 说明 |
|------|------|
| **文件大小阈值** | 每个上下文文件的大小基于 Claude 质量退化临界点设计 |
| **渐进式披露** | orchestrator 先加载 `PROJECT.md` + `STATE.md`，再按需加载 `CONTEXT.md`/`RESEARCH.md` |
| **子代理隔离** | 每个计划执行都在独立子代理中进行，主会话保持响应速度 |
| **上下文监控** | `hooks/gsd-context-monitor.js` 在 60% 警告、70% 临界 |
| **跨会话记忆** | `/gsd-thread` 将知识存入 `.planning/threads/`，跨会话保留 |
| **会话恢复** | `/gsd-pause-work` 写入 `HANDOFF.json`，`/gsd-resume-work` 恢复 |

### 4.4 命令体系总览

GSD 提供了 67 个斜杠命令，按功能域分组：

| 功能域 | 命令数 | 代表命令 |

| **阶段管理** | 5 | `phase --insert/--edit/--remove`, `phase --audit`, `phase --fix` |

## 五、使用方式

### 5.1 安装步骤

**标准安装（交互式）**：
```bash
npx get-shit-done-cc@latest
```
安装器会提示选择运行时（14+ 选项）和安装位置（全局/本地）。

**非交互安装（CI/Docker）**：
```bash
# Claude Code 全局安装
npx get-shit-done-cc --claude --global

# Codex 本地安装
npx get-shit-done-cc --codex --local

# 所有运行时
npx get-shit-done-cc --all --global
```

**基于源码安装**：
```bash
git clone https://github.com/gsd-build/get-shit-done.git
cd get-shit-done
node bin/install.js --claude --local
```

**最小化安装**（仅核心技能，节省 ~94% token）：
```bash
npx get-shit-done-cc --minimal
```

### 5.2 初始化配置

安装后，在项目目录运行：
```
/gsd-new-project
```
系统会进行多轮提问理解你的想法，然后并行拉起研究代理，生成 `PROJECT.md`、`REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md`。

对有现有代码库的项目，先运行：
```
/gsd-map-codebase
```
然后 `new-project` 会真正理解你的代码库，提问聚焦于新增部分。

## 六、技术细节

### 6.1 安装器：多运行时适配

`bin/install.js`（11,376 行，474KB）是项目最大的单文件，处理 14+ 运行时的安装逻辑：

- **Claude Code**：写入 `commands/gsd/*.md` + `skills/gsd-*/SKILL.md` + `agents/gsd-*.md` + `hooks/gsd-*.sh`
- **Codex**：写入 `skills/gsd-*/SKILL.md` + `config.toml`（含 sandbox 配置和 hooks）
- **Copilot**：写入 `.github/copilot-instructions.md`（带 GSD 标记块）
- **Cline**：写入 `.clinerules`
- **其他运行时**：各有专用安装路径（Gemini 的 `~/.gemini/`、OpenCode 的 `~/.config/opencode/`、Kilo 的 `~/.config/kilo/` 等）

安装器处理的关键技术细节：
- **命名空间转换**：`/gsd:cmd` → `/gsd-cmd` 的连字符规范化（`scripts/fix-slash-commands.cjs`）
- **重复安装安全**：通过 ownership marker 检测已有 GSD 文件，避免破坏用户自定义
- **卸载支持**：`--uninstall` 标志可完整移除所有文件

### 6.2 同步运行时桥

GSD 面临一个核心工程问题：CJS 核心库是同步的（被 100+ 消费者和 21 个测试文件依赖），但 SDK 使用异步 API（`@anthropic-ai/claude-agent-sdk`）。

**解决方案**（来自 `CONTEXT.md` 的 `Sync Runtime Bridge Module` 说明）：

1. `sdk/src/runtime-bridge-sync/index.ts` 使用 `synckit` 在 Worker 线程中运行 SDK
2. `synckit` 基于 `SharedArrayBuffer` + `Atomics.wait` 实现 CJS 侧的同步等待
3. 首次调用成本 ~80ms（Worker 启动），稳态 ~0.1ms/次
4. 桥接层将异步异常映射为类型化结果 `{ ok: true, data } | { ok: false, exitCode, errorKind, ... }`
5. 子进程回退被禁用——未知命令直接返回 `unknown_command` 而不 spawn `gsd-sdk`

### 6.3 命令路由中枢

`get-shit-done/bin/lib/command-routing-hub.cjs` 是架构演化的关键里程碑（ADR-0012）：

```javascript
// 接口抽象
hub = createHub({ mode: 'sdk' | 'cjs', sdkLoader, cjsRegistry, manifest });
result = hub.dispatch({ family, subcommand, args, cwd, raw });
// Result = { ok: true, data } | { ok: false, errorKind, message }
```

关键设计决策：
- **模式在构造时锁定**：SDK 崩溃 → `SdkDispatchFailed`，不回退到 CJS（避免透明降级隐藏 bug）
- **6 种错误类型闭集**：`UnknownCommand`/`InvalidArgs`/`HandlerRefusal`/`HandlerFailure`/`SdkLoadFailed`/`SdkDispatchFailed`
- **永不抛异常**：纯结果返回，调用方负责转译为 `output()`/`error()`

首个迁移案例是 `phase-command-router.cjs`（#3788），后续计划将 `state.*`/`verify.*`/`init.*` 等全部迁移到此中枢。

### 6.4 运行时能力面管理（Surface）

`get-shit-done/bin/lib/surface.cjs` 实现了 ADR-0011 定义的运行时能力面管理：

- 每个运行时的配置目录根有 `.gsd-surface.json` 标记文件
- 能力面计算：`baseProfile ∪ explicitAdds − disabledClusters − explicitRemoves`
- 支持 cluster 级别的批量禁用（如禁用整个 `gsd-review` cluster）
- 与安装时 profile 标记（`.gsd-profile`）独立，互不干扰
- 通过集群映射（`clusters.cjs`）实现传递闭包

这使 `/gsd-surface` 命令可以在不重装的情况下动态调整可用技能集。

### 6.5 上下文利用率守卫

`hooks/gsd-context-monitor.js` 提供两个阈值的上下文窗口监控：

- **60% 警告**："考虑 `/gsd-thread` 将信息外存"
- **70% 临界**："推理质量可能下降"

通过 `/gsd-health --context` 暴露为结构化命令，也在状态栏（`hooks/gsd-statusline.js`，22KB）中可视化展示。

### 6.6 测试策略

GSD 有极其严格的测试规范：

## 二、核心亮点（5个突出设计亮点）

### 亮点1：Anti-Rationalization（反借口头辩护）机制

这是该项目最独特的设计创新。每个 Skill 都包含一个 **"Common Rationalizations"（常见借口头辩护）** 表格，格式为两列：

| Rationalization（借口头辩护） | Reality（现实） |
|---|---|
| "I'll write tests after the code works" | You won't. And tests written after the fact test implementation, not behavior. |

这个设计直接针对 AI 智能体的核心弱点：**智能体擅长为自己的偷懒行为找到看似合理的借口**。项目将人类工程师在与 AI 协作中反复遇到的"借口模式"硬编码进 Skill，每个借口都配有事实性的反驳，从根本上阻止智能体绕过低质量步骤。

例如，`incremental-implementation` Skill 中就包含了 6 条反借口头辩护：
- "I'll test it all at the end" → Bugs compound
- "It's faster to do it all at once" → Feels faster until 500 lines break
- "These changes are too small to commit separately" → Small commits are free
- "I'll add the feature flag later" → If not complete, shouldn't be user-visible
- "This refactor is small enough to include" → Refactors mixed with features make both harder
- "Let me run the build command again just to be sure" → Re-running unchanged code adds no information

### 亮点2：Progressive Disclosure（渐进式信息披露）

Skill 采用分层加载策略，并非一次性将所有内容塞入智能体的上下文窗口：

```
SKILL.md (入口，150-390行)
  ├── YAML Frontmatter (name + description — 启动时注入系统提示词)
  ├── Overview & Workflow (核心流程)
  └── 引用 → references/ (仅按需加载)
```

这种设计使得 23 个 Skill 可以共存而不会耗尽 token 预算。智能体启动时仅加载所有 Skill 的 description 行（用于触发匹配），完整的 SKILL.md 内容仅在需要时才会被读入上下文。

### 亮点3：Verification-as-Non-Negotiable（验证不可协商）

每个 Skill 都以 **Verification Checklist** 结尾，带有明确的证据要求：

```markdown
## Verification
- [ ] Every new behavior has a corresponding test
- [ ] All tests pass: `npm test`
- [ ] Bug fixes include a reproduction test that failed before the fix
- [ ] Test names describe the behavior being verified
- [ ] No tests were skipped or disabled
```

关键原则：**"Seems right" is never sufficient**（"看起来正确"永不充分）。所有验证项都要求实际证据（测试通过输出、Build 成功日志、浏览器截图等），彻底杜绝了智能体"看起来没问题"的偷懒路径。

### 亮点4：Three-Layer Architecture（三层架构：Skill / Persona / Command）

项目精心设计了三个独立层次，各司其职：

| 层级 | 是什么 | 职责 | 示例 |
|------|--------|------|------|
| **Skill** | 工作流（步骤 + 退出标准） | *How* — 怎么做 | `code-review-and-quality` |
| **Persona** | 角色（视角 + 输出格式） | *Who* — 谁来做 | `code-reviewer` (Staff Engineer) |
| **Command** | 用户入口（编排层） | *When* — 何时做 | `/ship` (并行扇出) |

三者之间有严格的组合规则：
- **Persona 不能调用其他 Persona**（编排是 Command 的职责）
- Skill 可以在 Persona 内部被调用
- `/ship` 命令使用**并行扇出（fan-out）+ 合并（merge）**模式，同时调用三个 Persona

这种分层避免了"元编排器调用另一个编排器"的架构反模式（anti-pattern），在文档中明确规定并问责。

### 亮点5：Skill 验证流水线（CI Validator）

项目包含一个 220 行的 **JavaScript 校验脚本**（`scripts/validate-skills.js`），在 CI 中自动执行，对每个 Skill 进行检查：

- **硬性错误（阻塞 CI）：** SKILL.md 存在性、YAML frontmatter 完整性、name 与目录名一致性、description ≤ 1024 字符、必需章节完整性
- **软性警告（不阻塞 CI）：** 跨 Skill 引用是否指向存在的 Skill

关键设计细节：Skill 无法自行声明豁免——豁免名单硬编码在 `validate-skills.js` 的 `SECTION_EXEMPT_SKILLS` 对象中，防止贡献者绕过校验器。

### 3.2 Skill 格式规范（SKILL.md Anatomy）

每个 Skill 遵循统一的结构规范，定义在 `docs/skill-anatomy.md` 中：

```markdown
---
name: skill-name-with-hyphens          # 必须：小写连字符，与目录名一致
description: Guides agents through...  # 必须：做什么 + Use when 触发条件
---

# Skill Title

## Overview                              # 电梯演讲：做什么 + 为什么重要

## When to Use                           # 触发条件 + 排除条件
- 正向触发
- 反向排除

## Process / Workflow / Core Process     # 步骤化工作流（可含 ASCII 流程图）

## Common Rationalizations               # 反借口头辩护表
| Rationalization | Reality |

## Red Flags                             # 违规信号

## Verification                          # 退出标准（带证据要求的 checklist）
- [ ] 条件1
- [ ] 条件2
```

**命名约定：**
- Skill 目录：`lowercase-hyphen-separated`（kebab-case）
- Skill 文件：固定为 `SKILL.md`（全大写）
- 辅助文件：`lowercase-hyphen-separated.md`
- 参考资料：统一放在 `references/`，而非 Skill 目录内

**编写原则（6条）：**

1. **Process over knowledge**（流程优于知识）—— Skill 是工作流，不是参考手册
2. **Specific over general**（具体优于笼统）—— "Run `npm test`" 优于 "verify the tests"
3. **Evidence over assumption**（证据优于假设）—— 每个验证项需要实际证据
4. **Anti-rationalization**（反借口头辩护）—— 每个可跳过的步骤都需反驳论据
5. **Progressive disclosure**（渐进式披露）—— 主线精简，参考资料按需加载
6. **Token-conscious**（Token 经济）—— 每段内容都需证明其存在的必要性

## 四、功能体系（6 阶段 23 个 Skill 全览）

### 4.1 Meta 阶段（1 个）

| Skill | 功能 | 行数 |
|-------|------|------|
| `using-agent-skills` | 元 Skill：Skill 发现决策树 + 核心运行行为（6条不可协商规则）。提供 ASCII 决策树将任务路由到正确的 Skill。 | 186 |

元 Skill 定义了 6 条**全局行为准则**（跨越所有 Skill）：
1. **Surface Assumptions** — 实现前显式列出假设
2. **Manage Confusion** — 遇到矛盾时 STOP，明确提问
3. **Push Back When Warranted** — 不做"应声虫"，对有问题的方法提出异议
4. **Enforce Simplicity** — 主动抵抗过度复杂化
5. **Maintain Scope Discipline** — 只碰被要求触碰的代码
6. **Verify, Don't Assume** — 证据而非假设

### 4.2 Define 阶段（3 个）—— 明确要构建什么

| Skill | 功能 | 核心流程 | 行数 |
|-------|------|----------|------|
| `interview-me` | 通过"一次一个问题"的访谈，在计划/规格/代码存在**之前**，提取用户真正想要的（而非他们认为应该想要的），直到 ≥95% 置信度 | Hypothesis → One Q at a time + Guess → Listen for "want vs should-want" → Restate intent | 225 |
| `idea-refine` | 结构化发散/收敛思维，将模糊想法转化为具体方案 | 发散探索 → 收敛筛选 → 落地提案 | 178 |
| `spec-driven-development` | 编写 PRD（目标、命令、结构、代码风格、测试策略、边界） | SPECIFY → PLAN → TASKS → IMPLEMENT（每个 Gate 须人工审批） | 200 |

`interview-me` 的设计尤为精巧：
- 要求智能体在每个问题前给出**假设 + 置信度数字**（强制诚实）
- 每次只问一个问题（避免批量提问导致肤浅回答）
- 识别"sounds-like-right-answer"信号（用户说"应该"而非"想要"）

`spec-driven-development` 采用 4 阶段门控流程，与 Google 工程实践中的 Design Doc 文化一脉相承，包含三层边界系统：
- **Always:** 提交前跑测试、遵循命名约定、验证输入
- **Ask first:** 数据库 schema 变更、添加依赖、修改 CI 配置
- **Never:** 提交 secrets、编辑 vendor 目录、未经批准删除失败测试

### 4.3 Plan 阶段（1 个）

| Skill | 功能 | 行数 |
|-------|------|------|
| `planning-and-task-breakdown` | 将 Spec 分解为小型的、可验证的任务，定义验收标准与依赖排序 | 223 |

每个任务遵循模板：`[ ] Task: 描述 → Acceptance: 验收条件 → Verify: 验证命令 → Files: 涉及文件`

核心约束：单个任务不应涉及超过 5 个文件的修改。

### 4.4 Build 阶段（7 个）—— 编写代码

| Skill | 功能 | 关键设计 | 行数 |
|-------|------|----------|------|
| `incremental-implementation` | 薄垂直切片式实现：Implement → Test → Verify → Commit → Next | 5 条规则（简易优先、范围纪律、一次一事、保持可编译、功能开关）、3 种切片策略 | 245 |
| `test-driven-development` | Red-Green-Refactor 循环 + Prove-It Pattern（bug修复） | 测试金字塔（80/15/5）、测试粒度（Small/Medium/Large）、Beyonce Rule、DAMP over DRY | 383 |
| `context-engineering` | 为智能体提供"在正确时间看到正确信息"的上下文管理 | 5 层上下文层次（Rules → Spec → Source → Error → History） | 289 |
| `source-driven-development` | 所有框架决策必须由官方文档支撑 | DETECT → FETCH → IMPLEMENT → CITE（检测版本→拉取文档→实现→标注来源） | 194 |
| `doubt-driven-development` | 对每个非平凡决策执行对抗性新鲜上下文审查 | CLAIM → EXTRACT → DOUBT → RECONCILE → STOP（可跨模型升级） | 243 |
| `frontend-ui-engineering` | 组件架构、设计系统、状态管理、响应式设计、WCAG 2.1 AA 可访问性 | 组合优于配置、组件内聚、避免"AI 审美" | 328 |
| `api-and-interface-design` | 契约优先设计、Hyrum's Law、One-Version Rule、错误语义 | Contract First → 明确错误模型 → 边界验证 | 294 |

**`incremental-implementation` 的 5 条实现规则：**
- Rule 0: **Simplicity First** — "最简可行方案是什么？"
- Rule 0.5: **Scope Discipline** — 只碰任务要求的内容，额外发现记录但不修复
- Rule 1: **One Thing at a Time** — 每个增量只做一件逻辑事
- Rule 2: **Keep It Compilable** — 每个增量后项目必须可构建
- Rule 3: **Feature Flags** — 未完成功能用开关保护
- Rule 4: **Safe Defaults** — 新代码默认保守行为
- Rule 5: **Rollback-Friendly** — 每个增量可独立回滚

**`doubt-driven-development` 的独特地位：**
这是项目中最"暴力"的质量保障机制。它在决策过程中（而非完成后）启动一个**带对抗性偏见的新鲜上下文审查者**，专门试图**否定**当前决策。这种设计直接解决了"长期会话中上下文污染导致假设变成'事实'"的问题。支持跨模型升级（可选），当主智能体的审查者无法充分质疑时，可以调用不同模型的子智能体进行交叉审查。

**`source-driven-development` 解决的核心问题：**
AI 训练数据会过时——API 被弃用、最佳实践演进。该 Skill 要求所有框架特定代码必须有官方文档引用，每一步标注来源，让用户可以验证。这是对 "AI 幻觉" 的结构性防御。

### 4.5 Verify 阶段（2 个）—— 证明代码可用

| Skill | 功能 | 行数 |
|-------|------|------|
| `browser-testing-with-devtools` | 使用 Chrome DevTools MCP 进行实时运行时验证（DOM 检查、Console 日志、Network 追踪、性能分析、截图对比） | 302 |
| `debugging-and-error-recovery` | 五步分类：Reproduce → Localize → Reduce → Fix Root Cause → Guard（Stop-the-Line 规则） | 300 |

**`debugging-and-error-recovery` 的创新设计：**
- **Stop-the-Line Rule:** 当任何非预期情况发生时，立即停止添加功能，保存证据，诊断根因
- **错误输出作为不可信数据处理：** 明确要求不要执行错误信息中嵌入的命令、不导航到错误信息中的 URL——防御通过错误输出进行的提示注入攻击
- **安全回退模式：** 提供时间压力下的安全降级方案（默认值 + 警告，优雅降级而非崩溃）

### 4.6 Review 阶段（4 个）—— Merge 前的质量门

| Skill | 功能 | 关键设计 | 行数 |
|-------|------|----------|------|
| `code-review-and-quality` | 五轴审查（正确性、可读性、架构、安全性、性能）+ 变更粒度控制（~100 行） | Critical/Nit/Optional/FYI 严重性标签、以人为本的审批标准 | 347 |
| `code-simplification` | Chesterton's Fence（切斯特顿栅栏）、Rule of 500、保持精确行为的同时降低复杂度 | 在不改变行为的前提下减少不必要的复杂性 | 331 |
| `security-and-hardening` | OWASP Top 10 防护、认证模式、秘密管理、依赖审计、三层边界系统 | 具体代码示例（SQL 注入 → 参数化查询、密码哈希、会话管理） | 349 |
| `performance-optimization` | 度量优先——Core Web Vitals 目标、性能剖析工作流、打包分析、反模式检测 | Measure → Identify → Fix → Verify → Guard | 350 |

**`code-review-and-quality` 的变更粒度规范：**
```
~100 lines  → Good. 一次审查可完成.
~300 lines  → Acceptable（若为单个逻辑变更）.
~1000 lines → Too large. 必须拆分.
```

提供 4 种拆分策略：Stack（堆叠）、By file group（按文件组）、Horizontal（水平）、Vertical（垂直）。

**审批标准：** "Approve when it definitely improves overall code health, even if it isn't perfect."（当变更明确改善整体代码健康度时批准，即使不完美。）—— 这借鉴了 Google Code Review 指南的核心精神。

**`security-and-hardening` 的三层边界系统：**
- **Always:** 验证所有外部输入、参数化查询、编码输出、HTTPS、密码哈希、安全头部
- **Ask First:** 新认证流、新敏感数据类别、新外部集成、CORS 变更、文件上传
- **Never:** 提交 secrets、记录敏感数据、信任客户端验证、使用 `eval()`/`innerHTML` 处理用户数据

**`shipping-and-launch` 的 Feature Flag 生命周期：**

## 七、技术细节

### 7.1 Skill 编写规范总结

**必需项：**
- `skills/<skill-name>/SKILL.md` 文件
- 有效的 YAML frontmatter，含 `name` 和 `description`
- `description` 必须包含"做什么"和"何时使用"两部分，最大 1024 字符

**推荐项：**
- 标准章节：Overview → When to Use → Process → Common Rationalizations → Red Flags → Verification
- 等效章节名也可接受（如 `How It Works`、`Workflow`、`Core Process`）
- 支持文件仅在内容超过 100 行时创建

**禁止项：**
- 不创建空的 `scripts/` 目录
- 不在 Skill 间重复内容——引用其他 Skill
- 不创建"建议式"的模糊 Skill——必须是可执行步骤

## 八、总结

### 项目定位的精妙之处

Agent Skills 解决了一个根本问题：**AI 智能体的默认路径是"最短路径"，而最短路径通常对应原型质量，而非生产质量。** 它不是靠"更好的提示词"来改善 AI 输出，而是通过**结构化的流程约束**、**反借口头辩护机制**和**不可协商的验证要求**，从根本上改变了智能体的行为模式。

从全栈开发者的视角，这个项目的最大价值在于：
1. **可组合性**——23 个 Skill 可以按需组合，适合从单文件修复到完整特性开发的任何场景
2. **工具无关性**——纯 Markdown 格式使其可适配任何 AI 编码工具
3. **工程纪律的可编码化**——将资深工程师的判断力编码为可复用的步骤级指令
4. **防御性设计**——对智能体"偷懒路径"的系统性认知和结构性防御

Matt Pocock 将该项目定位为**面向"真正工程实践"的 AI 代理技能集合**——不是 vibe coding，而是将数十年软件工程经验浓缩为可复用的 AI 指令。其核心理念是：小、易适配、可组合，适用于任何模型，不与特定代理平台深度绑定。

项目的上下文文档（`CONTEXT.md`）自述为：
> "A collection of agent skills (slash commands and behaviors) loaded by Claude Code."

## 二、核心亮点

### 2.1 基于经典软件工程理论的 AI 代理方法论

每个 skill 的设计都直接引用经典软件工程著作或思想：

| Skill | 理论基础 | 引文来源 |
|-------|---------|---------|
| `/grill-me` / `/grill-with-docs` | 沟通对齐 | *The Pragmatic Programmer*（David Thomas & Andrew Hunt） |
| `/grill-with-docs` 的共享语言机制 | 通用语言（Ubiquitous Language） | *Domain-Driven Design*（Eric Evans） |
| `/tdd` 的深度模块设计 | 深度模块理论 | *A Philosophy of Software Design*（John Ousterhout） |
| `/improve-codebase-architecture` | 投资每日设计 | *Extreme Programming Explained*（Kent Beck） |
| `/diagnose` 的 Seam 概念 | 接缝测试 | Michael Feathers 的 *Working Effectively with Legacy Code* |

这赋予了项目超越"提示词模板合集"的深度。

### 2.2 "共享语言"（Shared Language）——项目中最强大的设计

`/grill-with-docs` + `CONTEXT.md` + `ADR` 三位一体构成了一套**术语对齐系统**：

1. **CONTEXT.md**：项目级领域术语表，规定精确的术语并明确禁用词（`_Avoid_`）
2. **grill-with-docs**：在需求讨论中实时挑战模糊用语，发现冲突时当场更新术语表
3. **ADR（Architecture Decision Records）**：仅当决策同时满足三个条件时才记录——难以逆转、缺乏上下文时会令人困惑、经过真实权衡

实际效果：将 "a lesson inside a section of a course is made 'real'" 压缩为 "materialization cascade"——既减少了代理的 token 消耗，又使代码命名保持一致。

### 2.3 "渐进式依赖"（Progressive Dependency）架构

项目通过 ADR `0001` 明确区分了 skill 之间的两种依赖关系：

- **硬依赖（Hard dependency）**：`to-issues`、`to-prd`、`triage` 必须依赖 `setup-matt-pocock-skills` 提供的配置，否则输出完全错误
- **软依赖（Soft dependency）**：`diagnose`、`tdd`、`improve-codebase-architecture`、`zoom-out` 仅使用模糊引用（"the project's domain glossary"），文档缺失时也能降级运行

这种设计避免了"到处写 `run /setup first`"的模板化。

### 2.4 "反馈回路优先"（Feedback Loop First）的调试哲学

`/diagnose` skill 将调试方法系统化为 6 步循环，其中最核心的洞见是：**"构建一个快速、确定、可代理运行的 pass/fail 信号——这本身就是技能"**。它提供了 10 种构建反馈回路的方法（从单元测试到 git bisect 再到人机交互脚本），按优先级排序。强调"没有反馈回路就不许进入推测阶段"——这是一种可执行的纪律，而非空泛建议。

### 2.5 "垂直切片"（Vertical Slices）优于"水平切片"的任务分解

`/to-issues` 和 `/tdd` 都强制执行 tracer bullet 模式：

```
WRONG (horizontal):  所有测试 → 所有实现
RIGHT (vertical):    测试1→实现1 → 测试2→实现2 → ...
```

每个垂直切片穿过所有层的完整路径（schema → API → UI → tests），确保每个 issue 独立可演示。这直接解决了 AI 代理"一次写太多代码然后全坏"的典型失败模式。

### 3.3 Skill 内部格式规范

每个 skill 目录的标准结构：

```
skill-name/
├── SKILL.md          # 必需：主指令文件
├── REFERENCE.md      # 可选：详细参考文档（如 LANGUAGE.md）
├── EXAMPLES.md       # 可选：示例（如 tests.md）
└── scripts/          # 可选：可执行脚本
    └── helper.sh
```

#### SKILL.md 格式约定

```markdown
---
name: skill-name
description: >              # 代理唯一看到的文本，决定何时触发
  Brief description of capability.
  Use when [specific triggers].
disable-model-invocation: true   # 可选：禁止代理自动触发
argument-hint: "..."             # 可选：参数提示
---

# Skill Name

<what-to-do>                  # 可选：主要任务指令，XML 标签包装
[核心行动描述]
</what-to-do>

<supporting-info>             # 可选：辅助信息
[领域感知、会话规则、术语约束...]
</supporting-info>
```

**Description 设计要求**（来自 `/write-a-skill`）：
- 最长 1024 字符
- 第三人称
- 第一句：做什么
- 第二句："Use when [specific triggers]"
- 这是**模型看到的唯一信息**，用于判断是否加载该 skill

#### 关键设计决策

1. **渐进式信息披露（Progressive Disclosure）**：SKILL.md 主文件控制 <100 行，详细内容拆分到独立文件，引用不超过一层深度
2. **XML 标签分段**：`<what-to-do>` 和 `<supporting-info>` 用 XML 标签清晰分隔"做什么"与"为什么/怎么判断"
3. **脚本优于生成代码**：确定性操作使用预写脚本（如 `block-dangerous-git.sh`），节省 token 并提高可靠性
4. **禁用字段**：`disable-model-invocation: true` 阻止代理自动触发（如 `zoom-out` 仅由用户显式调用）

## 四、功能体系

### 4.1 Engineering（工程技能，共 10 个）

#### `/setup-matt-pocock-skills`——初始化配置
为每个仓库搭建三块配置：Issue Tracker（GitHub/Linear/本地 markdown）、Triage Labels（五个规范角色到实际 label 的映射）、Domain Docs（单/多上下文的 CONTEXT.md 布局）。将配置写入 `CLAUDE.md` 或 `AGENTS.md` 的 `## Agent skills` 块和 `docs/agents/` 目录。采用交互式三步询问流程。

#### `/grill-with-docs`——带文档的深度质询
对用户计划进行逐题"拷问"，每道题提供推荐答案，一次只问一个。质询过程同步更新 `CONTEXT.md`（术语表）和 ADR（架构决策记录）。特有机制：
- **术语挑战**：用户用词与现有术语冲突时立即指出
- **模糊语锐化**：将 vague/overloaded 术语替换为精确规范术语
- **具体场景推演**：发明边界场景迫使精确化
- **代码交叉验证**：检查代码是否与口头描述一致

#### `/grill-me`——轻量质询
`grill-with-docs` 的纯质询版本，不写任何文件。适用于非编码场景或快速对齐。

#### `/tdd`——测试驱动开发
完整的红-绿-重构循环。核心思想：测试应验证行为（通过公共接口），而非实现细节。严格要求垂直切片模式——一次一个测试→一次实现。捆绑了 5 个参考文档：`tests.md`（好/坏测试示例）、`mocking.md`（何时 mock，如何设计可 mock 接口）、`deep-modules.md`（深度模块理论）、`refactoring.md`（重构清单）、`interface-design.md`（可测试性接口设计）。强调"不要先写所有测试再写所有实现"（horizontal slice 反模式）。

#### `/diagnose`——规范调试
六阶段调试方法论：
1. **构建反馈回路**（核心：提供 10 种方案，从单元测试到 HITL shell 脚本）
2. **复现**（确认反馈回路产生用户描述的相同故障）
3. **假设**（3-5 个可证伪假设，展示给用户评估前检查）
4. **测量**（每条探测映射到具体假设，用 `[DEBUG-xxx]` 标签便于清理）
5. **修复 + 回归测试**（修复前写回归测试，仅在存在"正确的 seam"时）
6. **清理 + 事后反思**（删除所有调试标记，提出架构改进建议）

#### `/triage`——Issue 分诊
通过状态机驱动 issue 流转。两个分类角色（`bug`/`enhancement`）和五个状态角色（`needs-triage` → `needs-info` → `ready-for-agent` → `ready-for-human` → `wontfix`）。关键流程：
- 展示"需要关注"的三个桶（未标记/待评估/待重评估）
- bug 类 issue 先尝试复现再进行 grilling
- 为 `ready-for-agent` 的 issue 撰写 Agent Brief（要求耐久性而非精度：不引用文件路径和行号，描述接口和行为合同）
- `wontfix` 的 enhancement 写入 `.out-of-scope/` 知识库

#### `/to-prd`——生成 PRD
将当前对话上下文合成为 PRD 并发布到 issue tracker。不采访用户——仅综合已知信息。PRD 模板包含：问题陈述、解决方案、用户故事（As a/an <actor>, I want...）、实现决策（不包含文件路径/代码片段）、测试决策、Out of Scope。

#### `/to-issues`——任务分解
将计划/PRD 分解为独立的垂直切片 issue。每个切片是一个 tracer bullet（穿透所有集成层端到端）。区分为 HITL（需人交互）和 AFK（agent 可独立完成）。发布前让用户审核粒度、依赖关系和 HITL/AFK 分类。

#### `/zoom-out`——代码概览
让代理上升一个抽象层次，用领域术语描述相关模块和调用者的全景图。极简实现（仅 7 行），`disable-model-invocation: true`，纯用户触发。

#### `/improve-codebase-architecture`——架构改善
发现并可视化"深化机会"（将浅层模块变为深层模块）。流程：
1. **探索**：通过"删除测试"识别浅层模块
2. **可视化报告**：生成自包含 HTML 文件，使用 Tailwind CDN + Mermaid 图表（before/after 对比），按推荐强度（Strong/Worth exploring/Speculative）分类
3. **交互深化**：对选定候选进行 grilling，同步更新 `CONTEXT.md` 和 ADR

使用严格的架构词汇表（来自 `LANGUAGE.md`）：Module、Interface、Implementation、Depth、Seam、Adapter、Leverage、Locality。

#### `/prototype`——快速原型
双分支原型系统：
- **LOGIC 分支**：轻量 TUI 终端应用，让用户手动驱动状态机（纯 reducer/状态机/纯函数 + TUI shell）
- **UI 分支**：单一路由上生成 3-5 个结构完全不同的 UI 变体，通过 `?variant=` 查询参数切换，底部浮动切换条

所有原型标记为 throwaway，规则：一条命令运行、不持久化、不抛光、表面状态、完成即删除或吸收。

### 4.2 Productivity（效率技能，共 4 个）

#### `/caveman`——超压缩模式
将 token 使用量降低约 75%。删除冠词、填充词、客套语，使用缩写，保留所有技术精确性。特殊例外：安全警告、不可逆操作、多步序列需自动解除压缩。

#### `/grill-me`——纯质询
不写任何文件，纯粹逐题深入采访。（详见 engineering 章节）

#### `/handoff`——会话交接
将当前对话压缩为交接文档，保存到 OS 临时目录。包含"建议使用哪些 skills"部分。不重复已存在于 PRD/ADR/issue 中的内容。自动脱敏（API keys、密码、PII）。

#### `/write-a-skill`——编写新技能
提供 skill 创作指南：结构规范、description 编写要求（"这是模型看到的唯一信息"）、何时添加脚本、何时拆分文件、审查清单。

### 4.3 Misc（杂项技能，共 4 个）

| Skill | 功能 |
|-------|------|
| `git-guardrails-claude-code` | 通过 Claude Code PreToolUse hook 阻止危险 git 命令（push, reset --hard, clean, branch -D 等），包含可执行 bash 脚本 |
| `migrate-to-shoehorn` | 将测试文件中的 `as` 类型断言迁移到 `@total-typescript/shoehorn` 的 `fromPartial()`/`fromAny()` |
| `scaffold-exercises` | 为 Total TypeScript 课程创建练习目录结构（sections/problems/solutions/explainers），集成 lint 检查 |
| `setup-pre-commit` | 配置 Husky + lint-staged + Prettier + typecheck + test 的 pre-commit hooks |

### 4.4 Deprecated（已弃用，4 个）

被替代或不再使用的技能，反映了项目的演化：
- `design-an-interface` → 被 `improve-codebase-architecture/INTERFACE-DESIGN.md` 替代

- `request-refactor-plan` → 被 `/improve-codebase-architecture` 替代

### 4.5 In-Progress（开发中，4 个）

- `writing-beats`：文章写作 beat-by-beat 结构

## 六、技术细节

### 6.1 Skill 编写规范

**文件组织**：
- 主文件 `SKILL.md` 控制在 100 行以内
- 内容超过 100 行时分拆到独立文件
- 引用只向下一层深度（`[REFERENCE.md](./REFERENCE.md)`）

**Description 规范**（来自 `/write-a-skill`）：
- 最长 1024 字符，第三人称
- 第一句：能力描述；第二句：触发条件（"Use when..."）
- 这是代理在选择加载哪个 skill 时唯一看到的文本

**Frontmatter 字段**：
| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | ✅ | skill 唯一标识 |
| `description` | ✅ | 触发匹配文本 |
| `disable-model-invocation` | ❌ | 禁止代理自动调用 |
| `argument-hint` | ❌ | 参数提示文本 |

## 二、核心亮点（5 个突出设计亮点）

### 2.1 哨兵注释驱动的安全卸载机制（v1.2.1+）

安装时在 `CLAUDE.md`、`HERMES.md`、`GEMINI.md` 等 bootstrap 文件中注入的内容会被 `<!-- superpowers-zh:begin -->` / `<!-- superpowers-zh:end -->` 哨兵注释包裹。卸载时按哨兵精确切除，**绝不会误删用户内容**。

```javascript
const SENTINEL_BEGIN = '<!-- superpowers-zh:begin (do not edit between these markers) -->';
const SENTINEL_END = '<!-- superpowers-zh:end -->';
```

这是对用户文件安全性的深度考量，体现了工程成熟度。

### 2.2 四维度全量质控审计系统

`scripts/audit.sh`（254 行）实现了工业级的质量保障体系，在每次 PR 和 push to main 时自动执行：

| 审计维度 | 检查内容 | 阻塞性 |
|----------|---------|--------|
| Category 1: 静态校验 | JSON parse / SKILL.md frontmatter 完整性 / symlink 有效性 / hook 可执行权限 | P0 |
| Category 2: Installer 功能 | 18 款工具的安装 → 幂等 → 卸载 完整回归 | P0 |
| Category 3: 上游对齐 | hooks 3 文件 + brainstorm scripts 3 文件 + 14 翻译 skill 结构层级 diff | P1 |
| Category 4: 交叉引用 | README → docs 链接 / skill 间 `superpowers:xxx` 引用 / 装后路径完整性 | P0 |

### 2.3 手动调用隔离设计——避免污染上游自动调度

4 个 `chinese-*` 技能被设计为**手动调用**（需用户显式输入 `/chinese-code-review` 等），不会自动触发。这是经过深思熟虑的架构决策：如果 `chinese-code-review` 自动触发，可能与 `requesting-code-review` 的自动调度冲突，造成 AI 行为混乱。手动调用使它们成为「参考资料」，而非干扰性工作流。

```yaml
# chinese-code-review/SKILL.md 的 frontmatter
description: 中文 review 沟通参考……仅在用户显式 /chinese-code-review 时调用，不要根据上下文自动触发。
```

### 2.4 全链路上游漂移检测与主动同步

项目不是简单地 fork 后放任自流，而是通过 CI workflow 主动监控上游 `obra/superpowers` 的变更：

- `git fetch upstream main --depth=50` → 逐文件 diff hooks、scripts、14 个翻译 skill 的 header 结构层级
- `scripts/audit.sh` 对每个翻译 skill 做 header 数对比（允许 ±3 波动，超过则报警）
- Release notes 清晰记录每次上游同步的具体文件和行数


安装器通过检测项目中的特征目录/文件（如 `.claude`、`.cursor`、`.hermes`、`HERMES.md` 等）自动识别工具。支持 18 款工具的 18 条安装路径，且支持 `--uninstall` 清理。这是对上游「每种工具各自手动安装」体验的巨大改进。

- **自动并行化**：复杂任务自动分配到专业 agents
- **持续执行**：不完成不停止（Ralph 持久化模式）
- **成本优化**：智能模型路由节省 30-50% token 消耗

## 二、核心亮点（5 个突出设计）

### 亮点一：层次化的编排模式体系

OMC 不是简单地将 agents 列表扔给用户，而是设计了**层次嵌套**的编排模式，从轻到重递进：

```
ultrawork（并行引擎）
  └─ ralph（持久化包装器，含 ultrawork + reviewer 验证）
       └─ autopilot（全自主流水线，含 ralph + 5 阶段生命周期）
            └─ deep-interview → ralplan → autopilot（最完整的 3 阶段管道）
```

此外还有独立的 **Team** 模式（Claude Code 原生 team 系统）作为并行执行面，以及 **UltraQA**（QA 循环）、**Ultragoal**（目标工件持久化）等。这种层次设计使得同一个引擎可以满足从"快速并行修复"到"完整产品交付"的全频谱需求。

### 亮点二：专业 agents 的深度提示工程

19 个 agents 不仅仅是名称和简单描述，每个都包含**精密的提示工程**：

- **结构化角色定义**：`<Role>`、`<Why_This_Matters>`、`<Success_Criteria>`、`<Constraints>`、`<Investigation_Protocol>`
- **失败模式预判**：`<Failure_Modes_To_Avoid>` 明确列出常见的错误路径
- **正面/反面示例**：`<Examples>` 中的 `<Good>` / `<Bad>` 对比
- **多模式支持**：code-reviewer 同时支持标准 review、style review、performance review、quality strategy 四种模式
- **工具限制**：通过 `disallowedTools` 前端元数据 + prompt 内约束双重机制实现只读/读写分离
- **子代理委托协议**：agents 内部可委托其他 agents（如 executor 可 spawn architect 交叉验证）

### 亮点三：基于文件系统的 Team 状态机

Team 模式实现了一个**完整的分布式任务状态机**，全部基于文件系统（无外部依赖）：

- **任务生命周期**：`pending → blocked → in_progress → completed/failed`，带版本号乐观锁
- **租约机制**：任务声明（claim）带 15 分钟租约 + UUID token，防止 worker 僵死
- **工单去重**：worker canonicalization 自动合并重复 worker 条目
- **治理策略**：delegation_only、plan_approval_required、nested_teams_allowed 等可配置
- **事件日志**：`.omc/state/team/{name}/events.jsonl` 记录所有状态转换
- **邮箱通信**：inbox/outbox 文件实现 worker 间异步通信
- **阶段管道**：`team-plan → team-prd → team-exec → team-verify → team-fix` 含交接文档

### 亮点四：Hook 驱动的事件系统架构

11 种生命周期事件 + 20+ 个独立 hook 脚本，构成了 OMC 的**神经系统**：

| 事件 | 挂载的 hook 脚本 | 作用 |
|------|-----------------|------|
| UserPromptSubmit | keyword-detector, skill-injector | 魔法关键词检测、自动注入匹配的 skill |
| SessionStart | session-start, project-memory-session, wiki-session-start, setup-init/maintenance | 恢复持久化模式、项目记忆加载 |
| PreToolUse | pre-tool-enforcer | 工具调用前安全检查 |
| PostToolUse | post-tool-verifier, project-memory-posttool, post-tool-rules-injector | 工具调用后验证、规则注入 |
| Stop | context-guard-stop, persistent-mode, code-simplifier | 上下文守护、持久化模式强制继续 |
| SubagentStart/Stop | subagent-tracker, verify-deliverables | 子代理跟踪、交付物验证 |
| PreCompact | pre-compact, project-memory-precompact, wiki-pre-compact | 压缩前状态保存 |
| SessionEnd | session-end, wiki-session-end | 会话摘要、通知发送 |

这种 hook 架构使得 OMC 可以在不修改 Claude Code 源码的情况下，拦截和增强几乎所有关键行为。

## 六、技术细节

### 6.1 插件架构详解

#### 注册流程

1. Claude Code 启动时加载 `~/.claude/plugins/` 下的已安装插件
2. 读取 `.claude-plugin/plugin.json` 获取 skills、commands、MCP servers 列表
3. Skills 从 `skills/*/SKILL.md` 加载，每个 skill 有自己的目录，可包含子模板
4. Commands 从 `commands/*.md` 加载
5. MCP servers 通过 `.mcp.json` 注册
6. Hooks 从 `hooks/hooks.json` 加载，挂载到 Claude Code 生命周期事件

#### Skill 定义格式

Skills 使用 Markdown frontmatter + 结构化内容：

```yaml
---
name: ralph
description: Self-referential loop until task completion
argument-hint: "[--no-deslop] [--critic=architect|critic|codex] <task>"
level: 4
---
# 结构化 XML 指令
<Purpose>...</Purpose>
<Use_When>...</Use_When>
<Steps>...</Steps>
<Execution_Policy>...</Execution_Policy>
```

#### Agent 定义格式

Agents 也使用 Markdown frontmatter + XML 提示词：

```yaml
---
name: architect
description: Strategic Architecture & Debugging Advisor
model: opus
level: 3
disallowedTools: Write, Edit
---
<Agent_Prompt>
  <Role>...</Role>
  <Why_This_Matters>...</Why_This_Matters>
  <Success_Criteria>...</Success_Criteria>
  <Constraints>...</Constraints>
  <Investigation_Protocol>...</Investigation_Protocol>
  ...
</Agent_Prompt>
```

运行时通过 `src/agents/definitions.ts` 中的 `loadAgentPrompt()` 读取 .md 文件并解析为 TypeScript 对象。

### 6.2 Team 协作机制详解

Team 模式是 OMC 最复杂的子系统（`src/team/` 目录 60+ 个 TypeScript 文件，编译为 11,650 行的 `bridge/team.js`）。

#### 核心设计

**状态存储**：基于文件系统的 JSON 文件，位于 `.omc/state/team/{team_name}/`：
```
.omc/state/team/fix-ts-errors/
├── config.json          # Team 元数据 + workers 数组
├── manifest.json        # Team 清单
├── tasks/               # 任务文件 (task-1.json, task-2.json, ...)
├── workers/             # Worker 状态
│   └── worker-1/
│       ├── heartbeat.json   # 心跳
│       ├── inbox.md         # 接收消息
│       ├── outbox.jsonl     # 发送消息
│       ├── status.json      # 状态
│       ├── .ready           # 就绪标记
│       ├── AGENTS.md        # Worker 专属指令
│       └── identity.json    # 身份信息
├── mailbox/             # 跨 worker 邮箱
├── dispatch/            # 任务分发队列
├── approvals/           # 任务审批
├── events.jsonl         # 事件日志
├── phase-state.json     # 阶段状态
├── monitor-snapshot.json # 监控快照
└── summary-snapshot.json # 摘要快照
```

**任务声明协议**：
- 乐观锁：每个任务有 `version` 字段，声明时递增
- UUID claim token：防止意外释放
- 15 分钟租约：超时后其他 worker 可抢回
- 依赖检查：声明前验证 blockedBy 依赖已完成

**通信机制**：
- Worker → Leader：通过 `SendMessage` API（生成新会话轮次）
- Leader → Worker：写入 `inbox.md` 文件
- 事件广播：追加到 `events.jsonl`
- 共享邮箱：`mailbox/{worker}.json` + 文件锁

**生命周期**：
1. `TeamCreate` → leader 成为 `team-lead@name`
2. `TaskCreate x N` → 子任务分解
3. `Task(spawn) x N` → 并行 spawn workers
4. 监控循环：收消息 + TaskList 轮询
5. `shutdown_request` → `shutdown_response(approve: true)`
6. `TeamDelete` → 清理所有资源

### 6.3 Bridge 设计详解

Bridge 层是 OMC 与 Claude Code 之间的**连接组织**，由多个 CJS 打包文件组成：

| 文件 | 大小 | 入口 | 职责 |
|------|------|------|------|
| `bridge/cli.cjs` | 93,572 行 / 3.2MB | `omc` 命令 | CLI 主入口：所有 omc 子命令 |
| `bridge/team.js` | 11,650 行 / 439KB | Team 状态机 | 完整的 Team 操作库 |
| `bridge/mcp-server.cjs` | MCP 入口 | `.mcp.json` 注册 | 20+ 自定义工具 |
| `bridge/team-mcp.cjs` | Team MCP | Team 专用 MCP | Team 内部 MCP 功能 |
| `bridge/team-bridge.cjs` | Team 桥接 | Team 通信桥 | Team 通信桥接 |
| `bridge/runtime-cli.cjs` | 运行时 CLI | 独立运行时 | 独立运行时环境 |
| `bridge/gyoshu_bridge.py` | Python 桥 | Python 集成 | Python 功能桥接 |

**构建流程**（`npm run build`）：
```
1. tsc                    → dist/ (TypeScript 编译)
2. build-skill-bridge     → dist/hooks/skill-bridge.cjs
3. build-mcp-server       → dist/* + bridge/mcp-server.cjs
4. build-bridge-entry     → bridge/ 入口文件
5. compose-docs           → 文档聚合
6. build:runtime-cli      → bridge/runtime-cli.cjs
7. build:team-server      → Team 服务器
8. build:cli              → bridge/cli.cjs (主 CLI)
```

所有 bridge 文件使用 **esbuild** 打包为单文件 CJS，确保在 Node.js 环境中零依赖运行。

### 6.4 Magic Keyword 检测系统

`scripts/keyword-detector.mjs`（1,370 行）是 OMC 的**中央神经系统**。它在每次用户输入时运行（通过 `UserPromptSubmit` hook），执行：

1. **去除代码块**：在检测前剥离 markdown 代码块和行内代码
2. **信息性上下文过滤**：如果关键词出现在 "what is..."、"explain..." 等询问语境中，则不触发
3. **多关键词优先级**：cancelomc > ralph > autopilot > team > ultrawork > ccg > ralplan > deep-interview > ...
4. **状态感知**：检查当前是否有活动的模式状态，避免重复启动
5. **Ralph/Ralplan 执行门**：Ralph 激活时强制 `ralplan-first` 策略
6. **Skill 调用代币预算**：1200 字符限制的 skill 调用请求

### 6.5 持久化模式设计

Ralph 模式的持久化通过多层机制实现：

1. **Hook 拦截**：`persistent-mode.mjs` 在 `Stop` 事件中检查 PRD 完成状态
2. **迭代计数**：`incrementRalphIteration()` 追踪循环次数，有最大迭代限制
3. **状态文件**：`.omc/state/ralph-state.json` 保存活动状态
4. **PRD 驱动**：`prd.json` 定义用户故事 + 验收标准，逐个检查和标记
5. **进度记录**：`progress.txt` 记录每次迭代的实现细节和模式发现
6. **会话恢复**：`session-start.mjs` 在会话开始时检查并恢复持久化状态
7. **Tombstone 机制**：完成的工作流在 24 小时内被标记为 tombstone，防止自动重启

## 二、核心亮点

### 亮点 1：超大规模 MCP 工具体系（300+ 工具）

Ruflo 通过 MCP Server 暴露了 **300+ 个标准化工具**，分布在 20+ 个工具族中：`memory_*`、`agentdb_*`、`hooks_*`、`neural_*`、`swarm_*`、`agent_*`、`autopilot_*`、`browser_*`、`aidefence_*`、`federation_*`、`ruvllm_*` 等。这不是简单的数量堆砌——每个工具族都有独立的 ADR（架构决策记录）定义其契约，且有 smoke test 作为合约验证。工具族之间的依赖关系通过命名空间机制（如 `swarm-state`、`autopilot-patterns`）清晰解耦。

### 亮点 2：六层自学习闭环（4-Step Intelligence Pipeline）

项目设计了一个完整的四步智能管道：

```
RETRIEVE → JUDGE → DISTILL → CONSOLIDATE
（检索）   （判断）  （蒸馏）   （巩固）
```

- **RETRIEVE**：从 HNSW 向量索引拉取相关模式和过去轨迹
- **JUDGE**：对候选进行评分（成功/失败/部分）
- **DISTILL**：通过 MicroLoRA / SONA 自适应提取关键学习
- **CONSOLIDATE**：用 EWC++（弹性权重巩固）防止灾难性遗忘

高级能力包括：轨迹学习（trajectory learning）、ReasoningBank 模式存储、跨项目 IPFS 模式迁移、Thompson Sampling 成本感知模型路由。

### 亮点 3：零信任智能体联邦（Zero-Trust Agent Federation）

这是项目最具野心的特性之一——「给智能体配 Slack」：

- **mTLS + ed25519** 身份验证，无 API Key、无共享密钥

- **5 级信任模型**：UNTRUSTED → VERIFIED → ATTESTED → TRUSTED → PRIVILEGED
- **行为信任评分**：`0.4×success + 0.2×uptime + 0.2×threat + 0.2×integrity`
- **拜占庭共识（BFT）** 用于跨不信任节点的状态变更

### 3.3 架构设计特点

1. **DDD 分层**：`v3/src/` 按照领域驱动设计组织——`domain/`（领域模型）、`application/`（应用服务）、`infrastructure/`（基础设施），是教科书级的分层架构。

2. **MCP 优先**：所有功能通过 Model Context Protocol 暴露，CLI 和 Claude Code 是 MCP 工具的等同消费者。

3. **插件即合约**：每个插件的 ADR-0001 定义了 MCP 表面、命名空间约定、smoke test 即合约（"if the smoke passes, the contract is met"）。

4. **事件驱动**：`SwarmCoordinator` 使用 `EventEmitter` 发射 `agent:spawned`、`agent:terminated`、`agent:message` 等事件，与 hooks 系统深度集成。

5. **双模运行**：`@claude-flow/cli-core`（轻量，1.5s 冷启动）和 `@claude-flow/cli`（完整功能，35s 冷启动），插件脚本可以通过 `CLI_CORE=1` 环境变量切换。

## 四、功能体系

### 4.1 Agent Teams 机制

Ruflo 的 Agent Teams 体系是一个完整的智能体协作框架：

**Claude Code 原生层面**（无需 MCP）：
- `Task`：生成子智能体（支持 `name:` 用于寻址、`run_in_background: true` 用于并行）
- `SendMessage`：命名智能体间通信
- `TaskCreate/List/Get/Update/Output/Stop`：共享任务追踪
- `Monitor`：事件流实时监控
- `EnterWorktree/ExitWorktree`：每个智能体独立 Git worktree 隔离

**Settings.json 的 agentTeams 配置**：
```json
{
  "agentTeams": {
    "enabled": true,
    "teammateMode": "auto",
    "taskListEnabled": true,
    "mailboxEnabled": true,
    "coordination": {
      "autoAssignOnIdle": true,
      "trainPatternsOnComplete": true,
      "notifyLeadOnComplete": true,
      "sharedMemoryNamespace": "agent-teams"
    },
    "hooks": {
      "teammateIdle": { "enabled": true, "autoAssign": true },
      "taskCompleted": { "enabled": true, "trainPatterns": true }
    }
  }
}
```

**关键机制**：
- **空闲自动分配**：队友空闲时自动从任务列表取任务
- **完成自动训练**：任务完成时自动将成功模式写入 `agent-teams` 命名空间
- **Leader 通知**：子任务完成后通知 Leader
- **Hooks 驱动**：通过 `SubagentStop` hook 触发 `post-task` 处理

### 4.2 Swarm 协调机制

**六种拓扑**：
| 拓扑 | 适用场景 | 特点 |
|------|----------|------|
| `hierarchical` | 编码 swarm（推荐） | Queen-leader 集中协调，防漂移 |
| `mesh` | 对等协作 | 全连接，无单点 |
| `hierarchical-mesh` | 10+ agent 大团队 | Queen + 对等通信 |
| `ring` | 流水线任务 | 环形传递 |
| `star` | 中心化分发 | 所有节点连接中心 |
| `adaptive` | 动态调整 | 根据负载自动切换 |

**五种共识策略**：
- **Raft**：Leader 维持权威状态（默认，适合编码任务）
- **Byzantine**：容忍恶意节点（联邦场景）
- **Gossip**：最终一致性
- **CRDT**：无冲突数据类型
- **Quorum**：多数投票

**任务分发**：负载均衡 + 优先级排序 + 能力匹配 + 并发执行（`Promise.all`）。

**防漂移默认配置**（编码 swarm）：
```
topology: hierarchical
maxAgents: 6-8
strategy: specialized（明确角色，无重叠）
consensus: raft（Leader 保持权威状态）
memory: hybrid（SQLite + AgentDB）
```

### 4.3 Hooks 自动化系统

在 `.claude/settings.json` 中定义了 **7 类 hook 事件**：

| Hook 事件 | 触发时机 | 处理动作 |
|-----------|----------|----------|
| `PreToolUse` (Bash) | 执行 Bash 前 | `pre-bash` — 安全检查 |
| `PostToolUse` (Write/Edit) | 写文件后 | `post-edit` — 触发审计/索引 |
| `UserPromptSubmit` | 用户提交提示词 | `route` — 智能路由 + 记忆检索 |
| `SessionStart` | 会话开始 | `session-restore` + `auto-memory-hook` import |
| `SessionEnd` | 会话结束 | `session-end` — 状态持久化 |
| `Stop` | 模型停止 | `auto-memory-hook sync` — 同步记忆 |
| `PreCompact` (manual/auto) | 上下文压缩前 | `compact-manual/auto` + `session-end` |
| `SubagentStop` | 子智能体完成 | `post-task` — 学习 + 训练 |

所有 hook 通过 `node .claude/helpers/hook-handler.cjs` 分发，内部有：
- **5 秒全局超时保护**（防止 hook 挂起）
- **stdin 500ms 超时**（防止管道状态问题）
- **Intelligence 3 秒超时**
